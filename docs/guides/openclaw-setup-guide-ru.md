## Фаза 0: Что такое OpenClaw

### Что мы делаем и зачем

OpenClaw — это платформа для запуска персонального AI-ассистента на собственном сервере. Вы отправляете сообщение в Telegram или WhatsApp, оно попадает на ваш сервер, сервер пересылает запрос к AI-модели (Claude, GPT или другой), получает ответ и отправляет его обратно вам в мессенджер.

Зачем это нужно, если можно просто открыть ChatGPT в браузере? Три причины:

**Приватность.** Ваши сообщения хранятся на вашем сервере. Наружу уходят только запросы к AI-модели — и ничего больше. Никакой посредник не видит ваших переписок.

**Доступность.** Ассистент работает 24/7 прямо в привычном мессенджере. Не нужно открывать отдельный сайт или приложение.

**Мультиагентность.** На одном сервере можно запустить несколько AI-ассистентов с разными навыками. Один отвечает на вопросы, другой пишет код, третий читает и резюмирует документы — и каждый работает в изолированном контейнере (изолированная среда, как отдельная квартира в многоэтажке: соседи не могут зайти друг к другу).

### Схема

```
┌──────────┐         ┌──────────────────────────────────┐         ┌──────────────┐
│          │         │       Your VPS Server            │         │              │
│   You    │         │                                  │         │  AI Provider  │
│          │  mes-   │  ┌────────────────────────────┐  │ request │  (Claude,    │
│ Telegram ├────────►│  │     OpenClaw Gateway       │  ├────────►│   GPT, etc.) │
│   or     │  sage   │  │                            │  │         │              │
│ WhatsApp │◄────────┤  │  ┌────────┐  ┌────────┐   │  │◄────────┤              │
│          │ response│  │  │Agent 1 │  │Agent 2 │   │  │ response│              │
│          │         │  │  └────────┘  └────────┘   │  │         │              │
└──────────┘         │  └────────────────────────────┘  │         └──────────────┘
                     │                                  │
                     │  Messages stay here.             │
                     │  Only requests to the AI model   │
                     │  go outward.                     │
                     └──────────────────────────────────┘
```

**Gateway** (центральный узел) — основной процесс OpenClaw. Он постоянно работает на сервере, принимает сообщения из мессенджеров, направляет их нужному агенту и возвращает ответы.

**Агент** — AI-ассистент с назначенной ролью. Каждый агент — это комбинация AI-модели, набора инструкций и списка разрешённых инструментов. Агенты работают в изолированных Docker-контейнерах (программные «коробки», которые не дают одному агенту повлиять на другого или на сам сервер).

### Что вам понадобится

| Что | Зачем | Где взять |
|-----|-------|-----------|
| VPS-сервер (виртуальный сервер, работающий в дата-центре 24/7 — как компьютер, но в облаке) | Место, где будет жить OpenClaw | Hetzner, DigitalOcean, Aeza — см. Фазу 1 |
| Аккаунт в Telegram | Мессенджер, через который вы будете общаться с ассистентом | telegram.org |
| API-ключ AI-провайдера (секретный код для доступа к AI-модели — как пароль от сервиса) | Чтобы ваш сервер мог отправлять запросы к Claude или GPT | console.anthropic.com или platform.openai.com |
| Примерно 15–25 мин с установщиком, 30–40 мин вручную | На всю установку от начала до работающего ассистента | — |

### Принцип безопасности

OpenClaw спроектирован по принципу «максимальная изоляция по умолчанию»:

- **Ваши сообщения** хранятся только на вашем сервере.
- **Наружу** уходят только запросы к AI-модели (Claude, GPT) — и ничего больше.
- **Gateway** слушает только на localhost (только «внутри» сервера — к нему нельзя подключиться снаружи).
- **Каждый агент** работает в отдельном Docker-контейнере с минимальными привилегиями.
- **Удалённый доступ** — только через VPN (зашифрованный туннель, как секретный проход, которым пользуетесь только вы).

### Мультиагентная архитектура

На одном сервере можно запустить несколько AI-ассистентов, каждый со своей специализацией:

```
┌──────────────────── Single VPS Server ──────────────────┐
│                                                          │
│  ┌─ OpenClaw Gateway ────────────────────────────────┐  │
│  │                                                    │  │
│  │  ┌──────────────┐  ┌──────────────┐               │  │
│  │  │  Assistant    │  │   Coder      │               │  │
│  │  │  Claude       │  │   GPT-4o     │               │  │
│  │  │  General      │  │   Writes code│               │  │
│  │  │  questions    │  │   in sandbox │               │  │
│  │  └──────────────┘  └──────────────┘               │  │
│  │                                                    │  │
│  │  ┌──────────────┐  ┌──────────────┐               │  │
│  │  │  Reader       │  │  Researcher  │               │  │
│  │  │  Claude       │  │  GPT-4o      │               │  │
│  │  │  Document     │  │  Web search  │               │  │
│  │  │  summaries    │  │              │               │  │
│  │  └──────────────┘  └──────────────┘               │  │
│  │                                                    │  │
│  └────────────────────────────────────────────────────┘  │
│                                                          │
│  Telegram ──► Assistant (default)                        │
│               /coder ──► Coder                           │
│  WhatsApp ──► Assistant                                  │
└──────────────────────────────────────────────────────────┘
```

Каждый агент изолирован: у него свой API-ключ, свой набор инструментов и свой контейнер. Если один агент скомпрометирован — остальные не пострадают.

### Границы доверия

OpenClaw спроектирован как **персональный ассистент** — один Gateway = один пользователь. Любой авторизованный клиент получает полные операторские права над всем gateway. Разделения ролей внутри одного Gateway нет.

| Сценарий | Безопасность | Пояснение |
|----------|-------------|-----------|
| Один пользователь, несколько агентов | Безопасно | Стандартный режим работы. Docker-изоляция + политики инструментов ограничивают ущерб |
| Два доверенных члена семьи | Умеренный риск | Нет изоляции сессий — каждый видит историю другого |
| Команда/организация на одном Gateway | Небезопасно | Нет разделения ролей, нет раздельного аудита, учётные данные видны всем |

Если нужен доступ для нескольких пользователей — разворачивайте отдельные Gateway для каждого.

## Фаза 1: Получение сервера

### Что мы делаем и зачем

Нам нужен VPS (Virtual Private Server — виртуальный сервер в дата-центре, который работает круглосуточно, как удалённый компьютер, доступный через интернет). На нём будет жить OpenClaw.

Если у вас уже есть VPS с Ubuntu 22.04 или Debian 12 и вы можете подключиться к нему по SSH — переходите сразу к Фазе 2.

Если нет — следуйте шагам ниже.

### Основной путь: Автоматический установщик

Если у вас есть VPS с root-доступом по SSH, вы можете использовать автоматический установщик, чтобы выполнить Фазы 1–4 за один шаг.

**Предварительные требования:**
- VPS с root-доступом по SSH (Ubuntu 22.04 или Debian 12)
- Ключ авторизации Tailscale (получите на https://login.tailscale.com/admin/settings/keys)
- API-ключ Anthropic (с https://console.anthropic.com)
- Токен Telegram-бота (от @BotFather)

**Шаги:**

1. Подключитесь к серверу по SSH как root:

```bash
ssh root@your_ip_address
```

2. Запустите установщик:

```bash
curl -fsSL https://raw.githubusercontent.com/openclaw/openclaw/main/install.sh | bash
```

Установщик запросит у вас перечисленные выше данные и автоматически выполнит:
- Защиту сервера (Фаза 2)
- Установку и настройку OpenClaw (Фаза 3)
- Настройку первого агента (Фаза 4)

После завершения установщика переходите к [Фаза 5: Команда агентов](#phase-5-agent-team).

### Запасной вариант: Ручные шаги

Если вы предпочитаете понимать каждый шаг или установщик вам не подходит, следуйте ручным шагам ниже.

### Схема

```
┌──────────────┐                              ┌──────────────────┐
│              │        SSH connection         │                  │
│  Your        │  (encrypted management       │  VPS Server      │
│  computer    │   channel to the server)      │  Ubuntu 22.04    │
│              ├─────────────────────────────► │                  │
│  Terminal    │   ssh user@123.45.67.89      │  2 GB RAM        │
│              │                               │  20 GB SSD       │
└──────────────┘                              └──────────────────┘
```

### Минимальные требования к серверу

| Параметр | Минимум | Рекомендуется |
|----------|---------|---------------|
| RAM (оперативная память) | 2 ГБ | 4 ГБ |
| CPU (процессор) | 1 ядро | 2 ядра |
| Диск | 20 ГБ SSD | 40 ГБ SSD |
| ОС (операционная система) | Ubuntu 22.04 или Debian 12 | Ubuntu 22.04 — проще для начинающих |

### Выбор провайдера

| Провайдер | Мин. цена | Расположение | Особенности |
|-----------|----------|-------------|-------------|
| Hetzner | ~4 EUR/мес | Германия, Финляндия | Лучшее соотношение цена/качество, стабильная инфраструктура |
| DigitalOcean | ~6 USD/мес | США, Европа, Азия | Простой интерфейс, обширная документация |
| Aeza | ~200 руб/мес | Россия, Европа | Русскоязычная поддержка, оплата в рублях |

Для данного руководства подойдёт любой провайдер. Примеры ниже написаны в общей форме.

### Шаги

#### Шаг 1. Зарегистрируйтесь у выбранного провайдера

Перейдите на сайт провайдера и создайте аккаунт. Вам понадобятся электронная почта и способ оплаты.

- Hetzner: https://www.hetzner.com/cloud
- DigitalOcean: https://www.digitalocean.com
- Aeza: https://aeza.net

#### Шаг 2. Сгенерируйте SSH-ключ

SSH-ключ — это цифровой пропуск для входа на сервер. Вместо пароля (который можно подобрать) используется пара файлов: публичный ключ (замок, который вы ставите на сервер) и приватный ключ (ключ, который остаётся только у вас). Подобрать такой «ключ» практически невозможно.

Откройте терминал (программу для ввода команд) на вашем компьютере и выполните команду:

**macOS и Linux** — откройте приложение Terminal:

```bash
ssh-keygen -t ed25519 -C "openclaw-vps"
```

**Windows** — откройте Windows Terminal или PowerShell (нажмите Win, введите «PowerShell», откройте найденную программу):

```powershell
ssh-keygen -t ed25519 -C "openclaw-vps"
```

Нажимайте Enter на все запросы (путь по умолчанию, пустая парольная фраза на данный момент).

Ожидаемый вывод:

```
Generating public/private ed25519 key pair.
Enter file in which to save the key (/Users/yourname/.ssh/id_ed25519):
Enter passphrase (empty for no passphrase):
Enter same passphrase again:
Your identification has been saved in /Users/yourname/.ssh/id_ed25519
Your public key has been saved in /Users/yourname/.ssh/id_ed25519.pub
The key fingerprint is:
SHA256:xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx openclaw-vps
```

Теперь выведите публичный ключ, чтобы его можно было скопировать:

```bash
cat ~/.ssh/id_ed25519.pub
```

Ожидаемый вывод — строка вида:

```
ssh-ed25519 AAAAC3Nza... openclaw-vps
```

Скопируйте эту строку целиком. Она понадобится на следующем шаге.

#### Шаг 3. Создайте сервер

В панели управления провайдера создайте новый сервер со следующими параметрами:

- **ОС:** Ubuntu 22.04
- **Тариф:** минимум 2 ГБ RAM, 1 vCPU, 20 ГБ SSD
- **Регион:** ближайший к вам (для минимальной задержки)
- **SSH-ключ:** вставьте публичный ключ из предыдущего шага (строку `ssh-ed25519 AAAAC3Nza... openclaw-vps`)
- **Имя сервера:** любое, например `openclaw`

После создания провайдер покажет IP-адрес (набор цифр вида `123.45.67.89`) вашего сервера. Запишите его.

#### Шаг 4. Подключитесь к серверу

SSH (Secure Shell) — зашифрованное соединение для удалённого управления сервером. Вы вводите команды на своём компьютере, а они выполняются на сервере.

Подождите 1–2 минуты после создания сервера (ему нужно время для запуска) и выполните:

```bash
ssh root@your_ip_address
```

Замените `your_ip_address` на IP, который показал вам провайдер. Например: `ssh root@123.45.67.89`.

При первом подключении система спросит:

```
The authenticity of host '123.45.67.89' can't be established.
ED25519 key fingerprint is SHA256:xxxxxxxxxxx.
Are you sure you want to continue connecting (yes/no/[fingerprint])?
```

Введите `yes` и нажмите Enter. Это нормальный вопрос при первом подключении — система запоминает «отпечаток» сервера.

Ожидаемый вывод — командная строка сервера:

```
root@openclaw:~#
```

Вы внутри сервера.

#### Шаг 5. Обновите систему

Первое, что нужно сделать на свежем сервере — установить все обновления безопасности:

```bash
apt update && apt upgrade -y
```

`apt` — это менеджер пакетов (программа для установки и обновления ПО на Ubuntu/Debian). `update` скачивает список обновлений, `upgrade` устанавливает их. Флаг `-y` означает «автоматически отвечать „да" на все вопросы».

Ожидаемый вывод — список обновлённых пакетов, заканчивающийся чем-то вроде:

```
XX upgraded, XX newly installed, 0 to remove and 0 not upgraded.
```

#### Шаг 6. Создайте обычного пользователя

Сейчас вы работаете как `root` — пользователь с неограниченными привилегиями. Это как ходить по зданию с мастер-ключом от всех дверей: удобно, но опасно — любая ошибка или взлом могут повредить всю систему.

Принцип минимальных привилегий: для повседневной работы используйте обычного пользователя, а привилегии `root` запрашивайте только при необходимости (через команду `sudo`).

Создайте нового пользователя:

```bash
adduser openclaw
```

Система попросит задать пароль и заполнить информацию о пользователе. Задайте надёжный пароль; остальные поля можно оставить пустыми (нажимайте Enter).

Ожидаемый вывод:

```
Adding user `openclaw' ...
Adding new group `openclaw' (1001) ...
Adding new user `openclaw' (1001) with group `openclaw' ...
Creating home directory `/home/openclaw' ...
Copying files from `/etc/skel' ...
New password:
Retype new password:
passwd: password updated successfully
...
Is the information correct? [Y/n] Y
```

Дайте новому пользователю право выполнять команды от имени администратора через `sudo` (команда, которая позволяет обычному пользователю выполнить одну конкретную команду с привилегиями root — вместо того чтобы быть root всё время):

```bash
usermod -aG sudo openclaw
```

#### Шаг 7. Настройте SSH-доступ для нового пользователя

Скопируйте ваш SSH-ключ от пользователя root к новому пользователю, чтобы можно было заходить на сервер сразу под новой учётной записью:

```bash
mkdir -p /home/openclaw/.ssh
cp /root/.ssh/authorized_keys /home/openclaw/.ssh/authorized_keys
chown -R openclaw:openclaw /home/openclaw/.ssh
chmod 700 /home/openclaw/.ssh
chmod 600 /home/openclaw/.ssh/authorized_keys
```

Что делает каждая команда:
- `mkdir -p` — создаёт директорию `.ssh` в домашней папке нового пользователя
- `cp` — копирует файл авторизованных ключей от root к новому пользователю
- `chown -R` — передаёт владение файлами новому пользователю (иначе SSH откажет в доступе)
- `chmod 700` и `chmod 600` — устанавливают строгие права доступа к файлам (только владелец может читать и изменять)

#### Шаг 8. Отключитесь и подключитесь заново как новый пользователь

Выйдите с сервера:

```bash
exit
```

Подключитесь снова, но теперь как пользователь `openclaw`:

```bash
ssh openclaw@your_ip_address
```

Ожидаемый вывод:

```
openclaw@openclaw:~$
```

Обратите внимание: вместо `root@` теперь показывается `openclaw@` — вы вошли как обычный пользователь. Для команд, требующих привилегий администратора, используйте `sudo` перед командой.

### Проверка

Выполните следующую команду, чтобы убедиться, что всё работает:

```bash
whoami && sudo whoami
```

Ожидаемый вывод:

```
openclaw
root
```

Первая строка — ваш текущий пользователь (`openclaw`). Вторая подтверждает, что `sudo` работает (команда `whoami` была выполнена от имени `root`). При первом использовании `sudo` система запросит пароль пользователя `openclaw`.

Если вы видите обе строки — сервер готов. Переходите к Фазе 2.

### Частые проблемы

**«Connection refused» при попытке подключиться по SSH**

Сервер ещё не загрузился. Подождите 2–3 минуты после создания и попробуйте снова. Если не помогает — проверьте IP-адрес в панели управления провайдера.

**«Permission denied (publickey)» при подключении**

SSH-ключ не был добавлен при создании сервера или был добавлен неправильно. Решения:
- Убедитесь, что вы скопировали публичный ключ (файл с расширением `.pub`).
- В панели управления провайдера удалите сервер, заново добавьте SSH-ключ и создайте сервер снова.
- Если провайдер это позволяет — откройте веб-консоль сервера и добавьте ключ вручную.

**Не могу найти терминал на Windows**

Нажмите клавишу `Win` на клавиатуре, введите «PowerShell» и откройте найденную программу. Если у вас Windows 11 — можно также найти «Windows Terminal» (более удобный вариант). Все SSH-команды работают одинаково в обеих программах.

**«sudo: command not found» для нового пользователя**

В некоторых минимальных образах Debian `sudo` не предустановлен. Войдите как `root` и установите его:

```bash
ssh root@your_ip_address
apt install -y sudo
usermod -aG sudo openclaw
exit
```

Затем войдите снова как `openclaw`.

**Забыли пароль нового пользователя**

Войдите как `root` и сбросьте пароль:

```bash
ssh root@your_ip_address
passwd openclaw
exit
```

## Фаза 2: Защита сервера

> **Автоматически:** Установщик выполняет эту фазу автоматически. Если вы использовали установщик, переходите к Фазе 5.

### Что мы делаем и зачем

Сейчас ваш сервер — как дом с открытыми дверями: кто угодно может попытаться войти. В этой фазе мы выстраиваем пять уровней защиты, чтобы даже при прорыве одного уровня остальные продолжали работать. Это называется «эшелонированная оборона» — когда одна линия защиты падает, за ней стоит следующая.

### Схема

```
Internet
    │
    ▼
┌──────────────────────────────────────────────────────────────────┐
│  Layer 1: UFW firewall — blocks everything except SSH & Tailscale│
├──────────────────────────────────────────────────────────────────┤
│  Layer 2: Fail2ban — bans IP after 5 failed login attempts       │
├──────────────────────────────────────────────────────────────────┤
│  Layer 3: Tailscale VPN — private network, server is invisible   │
├──────────────────────────────────────────────────────────────────┤
│  Layer 4: SSH key-only — passwords completely disabled           │
├──────────────────────────────────────────────────────────────────┤
│  Layer 5: Auto-updates — security holes are patched automatically│
└──────────────────────────────────────────────────────────────────┘
    │
    ▼
Your server — secured
```

### Требования безопасности (ПРОЧИТАЙТЕ ПЕРЕД НАЧАЛОМ ФАЗЫ 2)

Прежде чем вносить какие-либо изменения в безопасность, подготовьте запасной выход:

1. **Узнайте URL веб-консоли вашего VPS-провайдера** — это ваш последний способ доступа, если вы заблокируете себя:
   - Hetzner: https://console.hetzner.cloud
   - DigitalOcean: https://cloud.digitalocean.com
   - Aeza: https://my.aeza.net
2. **Откройте веб-консоль в отдельной вкладке браузера ПРЯМО СЕЙЧАС**, прежде чем продолжить. Убедитесь, что у вас есть к ней доступ.
3. **ПРАВИЛО: НИКОГДА не закрывайте текущую SSH-сессию во время Фазы 2.** Все проверки нужно выполнять в НОВЫХ окнах терминала. Ваша исходная сессия — это страховочный канал.

#### Контрольные точки безопасности

| После шага | Проверка | Если не прошла |
|-----------|-------|-----------|
| UFW enable | Можете ли вы подключиться по SSH из НОВОГО терминала? | Выполните `sudo ufw disable` в ИСХОДНОМ терминале |
| SSH restart | Вход по ключу работает И пароль отклоняется? | Восстановите резервную копию в исходном терминале (см. инструкции ниже) |
| Before chattr | Можете ли вы подключиться по SSH с 2+ разных устройств/сетей? | Исправьте доступ, прежде чем блокировать конфигурацию |

### Шаги

#### Шаг 1. Настройка UFW firewall

Firewall — это охранник на входе в ваш сервер. Он решает, какой сетевой трафик пропускать, а какой блокировать. UFW (Uncomplicated Firewall) — это упрощённая утилита для управления этим охранником.

**1.1. Запретить все входящие соединения по умолчанию:**

```bash
sudo ufw default deny incoming
```

Эта команда говорит firewall: «блокируй абсолютно всё, что приходит снаружи, если я явно не разрешу». Это самое безопасное начальное состояние.

Ожидаемый вывод:
```
Default incoming policy changed to 'deny'
```

**1.2. Разрешить SSH-соединения:**

```bash
sudo ufw allow 22/tcp
```

Порт — это пронумерованная «дверь» на сервере. Порт 22 — стандартная дверь для SSH. Если вы не откроете его, вы не сможете подключиться к серверу после включения firewall.

Ожидаемый вывод:
```
Rules updated
Rules updated (v6)
```

**1.3. Разрешить Tailscale:**

```bash
sudo ufw allow 41641/udp
```

Порт 41641 нужен для Tailscale VPN, который мы настроим на шаге 3. UDP — это тип сетевого протокола (в отличие от TCP, он быстрее, но менее надёжен; для VPN это оптимальный выбор).

Ожидаемый вывод:
```
Rules updated
Rules updated (v6)
```

> **⚠️ КРИТИЧЕСКИ ВАЖНО: НЕ ЗАКРЫВАЙТЕ ТЕКУЩУЮ SSH-СЕССИЮ**
>
> После включения UFW проверьте подключение из **НОВОГО** окна терминала.
> Держите текущую сессию открытой как страховочный канал, пока не убедитесь, что доступ работает.

**1.4. Включить firewall:**

```bash
sudo ufw enable
```

Система запросит подтверждение — введите `y` и нажмите Enter.

Ожидаемый вывод:
```
Command may disrupt existing SSH connections. Proceed with operation (y|n)? y
Firewall is active and enabled on system startup
```

Firewall теперь будет автоматически запускаться при каждой перезагрузке сервера.

> **🛑 КОНТРОЛЬНАЯ ТОЧКА — Проверьте SSH-доступ**
>
> Откройте **новый терминал** и попробуйте подключиться к серверу по SSH:
> ```bash
> ssh openclaw@YOUR_SERVER_IP
> ```
> - ✅ Если работает: переходите к следующему шагу
> - ❌ Если не работает: выполните `sudo ufw disable` в **исходном** терминале и разберитесь с проблемой
>
> **НЕ продолжайте, пока эта проверка не пройдена.**

#### Шаг 2. Установка Fail2ban

Fail2ban — это автоматический блокировщик. Он следит за логами входа и, если кто-то пытается подобрать пароль несколько раз подряд, блокирует его IP-адрес (IP-адрес — уникальный номер каждого устройства в интернете, как почтовый индекс).

**2.1. Установка:**

```bash
sudo apt update && sudo apt install -y fail2ban
```

Ожидаемый вывод — длинный список строк, заканчивающийся:
```
Setting up fail2ban ...
```

**2.2. Создание файла конфигурации:**

Fail2ban хранит свои настройки в директории `/etc/fail2ban/`. Файл `jail.local` содержит ваши пользовательские правила, которые не будут перезаписаны при обновлении программы.

```bash
sudo tee /etc/fail2ban/jail.local << 'EOF'
[sshd]
enabled = true
port = ssh
filter = sshd
maxretry = 5
bantime = 3600
findtime = 600
EOF
```

Что означают эти настройки:
- `enabled = true` — защита SSH включена
- `maxretry = 5` — после 5 неудачных попыток входа IP блокируется
- `bantime = 3600` — блокировка на 3600 секунд (1 час)
- `findtime = 600` — 5 попыток должны произойти в течение 600 секунд (10 минут), чтобы сработала блокировка

**2.3. Запуск и включение автозапуска:**

```bash
sudo systemctl enable --now fail2ban
```

Команда `systemctl` управляет сервисами операционной системы. Флаг `enable` включает автозапуск при перезагрузке, а `--now` запускает сервис прямо сейчас.

Ожидаемый вывод:
```
Synchronizing state of fail2ban.service...
```

**2.4. Проверка работы Fail2ban:**

```bash
sudo fail2ban-client status sshd
```

Ожидаемый вывод:
```
Status for the jail: sshd
|- Filter
|  |- Currently failed: 0
|  `- Total failed:     0
`- Actions
   |- Currently banned: 0
   `- Total banned:     0
```

#### Шаг 3. Настройка Tailscale VPN

Вместо того чтобы ваш сервер был виден всему интернету, VPN (Virtual Private Network) создаёт приватный туннель. Представьте: сейчас ваш сервер стоит на оживлённой улице с табличкой «Я здесь». С VPN вы прячете его в закрытом дворе, и только ваши устройства имеют ключ от ворот.

Tailscale — самый простой VPN для нашей задачи. Он использует технологию WireGuard и не требует сложной настройки.

**3.1. Создание аккаунта Tailscale:**

Откройте https://login.tailscale.com/start в браузере и зарегистрируйтесь. Можно войти через аккаунт Google, GitHub или Microsoft.

**3.2. Установка Tailscale на VPS:**

```bash
curl -fsSL https://tailscale.com/install.sh | sh
```

Эта команда скачивает и запускает установщик. `curl` — утилита для скачивания файлов из интернета. Флаги `-fsSL` означают: не показывать прогресс-бар, но показывать ошибки и следовать перенаправлениям.

Ожидаемый вывод заканчивается строкой:
```
Installation complete.
```

**3.3. Подключение VPS к вашей сети Tailscale:**

```bash
sudo tailscale up
```

Команда выведет ссылку для авторизации:
```
To authenticate, visit:

    https://login.tailscale.com/a/abc123...
```

Откройте эту ссылку в браузере на вашем компьютере и подтвердите подключение.

**3.4. Узнайте Tailscale IP вашего сервера:**

```bash
tailscale ip -4
```

Ожидаемый вывод — IP-адрес вида:
```
100.x.y.z
```

Запишите этот адрес — вы будете использовать его для подключения к серверу через VPN.

**3.5. Установите Tailscale на ваше устройство (компьютер или телефон):**

- **macOS/Windows/Linux:** скачайте с https://tailscale.com/download
- **iOS:** App Store, поиск «Tailscale»
- **Android:** Google Play, поиск «Tailscale»

После установки войдите под тем же аккаунтом, что и на шаге 3.1.

**3.6. Проверка VPN-соединения:**

С вашего устройства (не с самого VPS) подключитесь к серверу через Tailscale IP:

```bash
ssh your_user@100.x.y.z
```

Замените `100.x.y.z` на адрес из шага 3.4, а `your_user` — на ваше имя пользователя на сервере.

Если соединение прошло успешно, вы достигли сервера через приватный туннель.

> **Проверьте Tailscale из нескольких сетей:**
> - Проверьте из домашнего WiFi
> - Проверьте через мобильный интернет (раздача с телефона)
> - Оба подключения должны работать, прежде чем вы будете полагаться на Tailscale как основной способ доступа

#### Шаг 4. Настройка автоматических обновлений безопасности

Каждый день в программном обеспечении обнаруживаются уязвимости. Автоматические обновления (unattended-upgrades) — это как автоматическая замена замков, если в старых обнаружен дефект.

**4.1. Установка:**

```bash
sudo apt install -y unattended-upgrades
```

**4.2. Включение автоматических обновлений:**

```bash
sudo dpkg-reconfigure -plow unattended-upgrades
```

Система спросит: «Автоматически скачивать и устанавливать стабильные обновления?» — выберите `Yes`.

**4.3. Проверка работы автообновлений:**

```bash
sudo unattended-upgrade --dry-run
```

Флаг `--dry-run` означает «пробный запуск» — команда покажет, что *было бы* обновлено, но ничего не изменит. Если команда завершится без ошибок, автообновления настроены правильно.

> **Примечание:** Unattended upgrades иногда могут перезапускать SSH. Это нормально, но может кратковременно прервать соединения.
> Следите за логом в `/var/log/unattended-upgrades/`, если столкнётесь с неожиданными отключениями.

#### Шаг 5. Отключение входа по паролю через SSH

Сейчас вы входите на сервер по SSH-ключу (настроено в Фазе 1). Но сервер по-прежнему принимает и парольный вход. Это лишняя дверь, которую нужно закрыть — пароли можно подобрать, ключи — практически невозможно.

**5.1. Откройте файл конфигурации SSH:**

```bash
sudo nano /etc/ssh/sshd_config
```

`nano` — простой текстовый редактор в терминале.

**5.2. Найдите и измените следующие строки:**

Используйте `Ctrl+W` для поиска. Найдите каждую из этих строк и убедитесь, что они выглядят именно так (без символа `#` в начале):

```
PasswordAuthentication no
PubkeyAuthentication yes
ChallengeResponseAuthentication no
```

Если строка начинается с `#` — удалите `#` (это символ комментария, отключающий настройку). Если значение `yes` вместо `no` — замените его.

Сохраните файл: `Ctrl+O`, затем `Enter`, затем `Ctrl+X` для выхода.

**5.3. Перезапустите сервис SSH:**

```bash
sudo systemctl restart sshd
```

Важно: не закрывайте текущую SSH-сессию! Откройте новый терминал и убедитесь, что можете подключиться по ключу. Если что-то пошло не так, в первом терминале вы сможете откатить изменения.

**5.4. Убедитесь, что вход по паролю отключён:**

В новом терминале попробуйте подключиться с принудительной парольной аутентификацией:

```bash
ssh -o PubkeyAuthentication=no your_user@your_server_IP
```

Ожидаемый вывод:
```
Permission denied (publickey).
```

Это означает, что сервер отклонил парольный доступ — именно то, что нам нужно.

> **🛑 КОНТРОЛЬНАЯ ТОЧКА — Проверьте конфигурацию SSH**
>
> Выполните ДВА теста из нового терминала:
>
> **Тест A — Аутентификация по ключу должна работать:**
> ```bash
> ssh openclaw@YOUR_SERVER_IP
> ```
> Ожидается: вход выполняется без запроса пароля
>
> **Тест B — Парольная аутентификация должна быть отклонена:**
> ```bash
> ssh -o PubkeyAuthentication=no openclaw@YOUR_SERVER_IP
> ```
> Ожидается: ошибка `Permission denied (publickey)`
>
> **Оба теста должны пройти, прежде чем продолжать. Если Тест A не прошёл, восстановите резервную копию SSH:**
> ```bash
> sudo chattr -i /etc/ssh/sshd_config  # if already locked
> sudo cp /etc/ssh/sshd_config.backup-openclaw-* /etc/ssh/sshd_config
> sudo systemctl restart sshd
> ```

---

> **⚠️ ПЕРЕД БЛОКИРОВКОЙ КОНФИГУРАЦИИ SSH**
>
> Проверьте SSH-доступ как минимум с 2 разных устройств или сетей, прежде чем делать конфигурацию неизменяемой.
> После `chattr +i` ЕДИНСТВЕННЫЙ способ изменить конфигурацию SSH — через веб-консоль VPS.

#### Шаг 6. Защита конфигурационных файлов от изменений

После того как настройки SSH зафиксированы, мы ставим на них «блокировку на уровне ядра» — атрибут `chattr +i` (immutable). Этот атрибут запрещает изменение файла даже пользователю root, как если бы вы повесили замок на дверь, ключ от которого нужно явно запрашивать каждый раз.

**6.1. Сделайте конфигурационные файлы неизменяемыми:**

```bash
sudo chattr +i /etc/ssh/sshd_config
sudo chattr +i /root/.ssh/authorized_keys
```

Теперь эти файлы нельзя изменить, удалить или переименовать — ни вручную, ни скриптом, ни вредоносным ПО.

**6.2. Как редактировать защищённые файлы**

Если в будущем вам понадобится изменить один из этих файлов, сначала снимите защиту, внесите изменения, затем верните защиту:

```bash
sudo chattr -i /etc/ssh/sshd_config
# ... make changes ...
sudo chattr +i /etc/ssh/sshd_config
```

**Важно:** Не используйте `chattr +i` на конфигурационных файлах OpenClaw (`openclaw.json`, `paired.json` и т.д.) — они должны быть доступны для записи во время работы программы. Для их защиты используйте `chmod 600` (доступ только для владельца) и проверку целостности через хеш-суммы (baseline).

### Частые проблемы

**Проблема: Потерян доступ к серверу после включения UFW**

Если вы случайно заблокировали собственный SSH-доступ:
1. Войдите на сервер через **веб-консоль** вашего VPS-провайдера (Hetzner, DigitalOcean и т.д. — ищите кнопку «Console» или «VNC» в панели управления)
2. Выполните:
```bash
sudo ufw disable
sudo ufw allow 22/tcp
sudo ufw enable
```

Веб-консоль работает напрямую, минуя сеть, поэтому firewall на неё не влияет.

**Проблема: Tailscale не подключается**

1. Убедитесь, что на обоих устройствах (VPS и ваш компьютер/телефон) вы вошли в один и тот же аккаунт Tailscale
2. На VPS проверьте статус:
```bash
sudo systemctl status tailscaled
```
Должно быть написано `active (running)`. Если нет:
```bash
sudo systemctl restart tailscaled
sudo tailscale up
```
3. Убедитесь, что порт 41641/udp открыт в UFW (шаг 1.3)

**Проблема: Fail2ban заблокировал мой IP**

Если вы несколько раз ввели неправильный пароль и вас заблокировали:
```bash
sudo fail2ban-client set sshd unbanip your_IP_address
```

Чтобы узнать, какие IP заблокированы:
```bash
sudo fail2ban-client status sshd
```

**Проблема: Не удаётся войти после отключения паролей**

Если парольный вход отключён, а ключ не работает:
1. Войдите через веб-консоль VPS-провайдера
2. Верните парольный вход:
```bash
sudo nano /etc/ssh/sshd_config
```
Измените `PasswordAuthentication no` на `PasswordAuthentication yes`, сохраните и выполните:
```bash
sudo systemctl restart sshd
```
3. Заново настройте SSH-ключ (по инструкции из Фазы 1), затем повторите шаг 5

Node.js — это среда выполнения JavaScript. OpenClaw написан на JavaScript/TypeScript, и для его работы необходим Node.js.


Ожидаемый вывод:

pnpm — это менеджер пакетов для Node.js, аналог apt, но для JavaScript-библиотек. OpenClaw распространяется через npm-реестр, и pnpm необходим для его установки.


`corepack` — встроенная утилита Node.js для управления менеджерами пакетов.

#### Шаг 3. Установка Docker CE

Docker — это платформа контейнеризации. Контейнер — как грузовой контейнер на корабле: программа внутри изолирована от всего остального на сервере. Если AI-агент выполнит что-то опасное, это произойдёт внутри контейнера и не затронет ваш сервер.

```bash
sudo apt install -y ca-certificates curl gnupg
sudo install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
sudo chmod a+r /etc/apt/keyrings/docker.gpg
```

Эти команды скачивают и устанавливают криптографический ключ Docker — он нужен, чтобы ваша система доверяла пакетам от Docker.

```bash
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] https://download.docker.com/linux/ubuntu \
  $(. /etc/os-release && echo "$VERSION_CODENAME") stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null
```

Эта команда добавляет репозиторий Docker в список источников пакетов вашей системы.

```bash
sudo apt update
sudo apt install -y docker-ce docker-ce-cli containerd.io docker-compose-plugin
```

Установка самого Docker и его компонентов.

```bash
sudo usermod -aG docker $USER
```

Добавляет вашего пользователя в группу `docker`, чтобы можно было запускать контейнеры без `sudo`. Чтобы изменение вступило в силу, нужно выйти и зайти снова:

```bash
exit
ssh your_user@100.x.y.z
```

Проверка:

```bash

```

Ожидаемый вывод содержит строку:
```
Hello from Docker!
This message shows that your installation appears to be working correctly.
```

#### Шаг 6. Применение защищённой конфигурации

Конфигурация (конфиг) — файл с настройками поведения программы. OpenClaw использует формат JSON5 (расширенная версия JSON, поддерживающая комментарии).

Откройте файл конфигурации:

```bash
nano ~/.openclaw/openclaw.json5
```

Замените содержимое на следующее:

```json5
{
  "$schema": "https://openclaw.ai/schemas/2024-11/config.json",
  "version": 1,

  "gateway": {
    "bind": "loopback",
    "port": 18789,
    "auth": {
      "mode": "token",
      "token": "INSERT_TOKEN_FROM_STEP_7_HERE"
    },
    "trustedProxies": [],
    "controlUi": {
      "dangerouslyDisableDeviceAuth": false
    }
  },

  "env": {},

  "session": {
    "dmScope": "per-channel-peer"
  },

  "agents": {
    "defaults": {
      "tools": {
        "profile": "minimal"
      },
      "sandbox": {
        "mode": "all",
        "workspaceAccess": "none"
      }
    }
  },

  "skills": {
    "autoInstall": false,
    "trustedPublishers": []
  },

  "logging": {
    "redactSensitive": "tools"
  },

  "browser": {
    "evaluateEnabled": false
  },

  "discovery": {
    "mdns": {
      "mode": "off"
    }
  },

  "plugins": {
    "enabled": false
  },

  "commands": {
    "config": false
  },

  "channels": {
    "telegram": {
      "linkPreview": false,
      "groups": {
        "*": {
          "requireMention": true,
          "tools": {
            "allow": ["read", "message"],
            "deny": ["exec", "write", "edit", "browser", "gateway", "nodes"]
          }
        }
      }
    }
  }
}
```

Что делает каждая настройка:

| Настройка | Значение | Зачем |
|-----------|----------|-------|
| `bind: "loopback"` | Gateway слушает только на `127.0.0.1` | Доступ только через Tailscale, не из открытого интернета |
| `auth.mode: "token"` | Вход по токену | Без токена никто не получит доступ к Gateway |
| `sandbox.mode: "all"` | Все инструменты агентов запускаются внутри Docker-контейнеров | Даже если агент сделает что-то опасное, он не выйдет за пределы контейнера |
| `sandbox.workspaceAccess: "none"` | У агентов нет доступа к файловой системе | Нет риска чтения или изменения файлов на сервере |
| `tools.profile: "minimal"` | Агенты получают минимальный набор инструментов | Меньше инструментов — меньше возможностей для злоупотребления |
| `redactSensitive: "tools"` | Чувствительные данные удаляются из логов | Токены и ключи не попадут в лог |
| `browser.evaluateEnabled: false` | Выполнение JavaScript в браузере отключено | Закрывает вектор инъекции кода |
| `discovery.mdns.mode: "off"` | Сервер не анонсирует себя в сети | VPS не нуждается в обнаружении в локальной сети |
| `plugins.enabled: false` | Плагины отключены | Сторонний код не будет выполняться, пока вы явно не разрешите |
| `commands.config: false` | Изменение конфигурации через чат запрещено | Настройки меняются только через терминал на сервере |
| `session.dmScope: "per-channel-peer"` | Отдельная сессия на каждый канал и собеседника | Предотвращает утечку контекста между пользователями и каналами |
| `skills.autoInstall: false` | Навыки не устанавливаются автоматически | Предотвращает атаки на цепочку поставок через вредоносные навыки |
| `channels.telegram.linkPreview: false` | Превью ссылок отключено | Блокирует утечку данных через специально сформированные URL |
| `channels.telegram.groups.*.requireMention: true` | Бот отвечает в группах только при упоминании | Предотвращает непреднамеренное использование инструментов участниками группы |

Не закрывайте файл — сначала сгенерируйте токен (шаг 7), вставьте его и только потом сохраняйте.

#### Шаг 6.5. Ограничение ресурсов Gateway

В Node.js (среда выполнения, на которой работает OpenClaw) существует известная проблема утечки памяти (Issue #13758 — Gateway может потреблять до 1,9 ГБ памяти после 13 часов непрерывной работы). Утечка памяти — это когда программа запрашивает память у системы, но никогда не возвращает её, как если бы вы брали тарелки из шкафа, но никогда не ставили их обратно — в конце концов шкаф опустеет.

Чтобы Gateway не «съел» всю память сервера, ограничим его ресурсы через systemd:

> **⚠️ Ограничения памяти НЕ являются опциональными.** Пропуск этой настройки может привести к аварийному исчерпанию памяти через 13 часов, что обрушит весь ваш VPS. Всегда настраивайте ограничения ресурсов, как показано выше.

```bash
# Add Node.js memory limit to systemd service
sudo mkdir -p /etc/systemd/system/openclaw.service.d
sudo tee /etc/systemd/system/openclaw.service.d/resources.conf << 'EOF'
[Service]
Environment=NODE_OPTIONS=--max-old-space-size=1536
MemoryHigh=1536M
MemoryMax=2G
Restart=always
RestartSec=5
EOF
sudo systemctl daemon-reload
sudo systemctl restart openclaw
```

Что делает каждая настройка:

| Настройка | Значение | Зачем |
|-----------|----------|-------|
| `--max-old-space-size=1536` | Ограничивает кучу Node.js до 1536 МБ | Предотвращает неконтролируемый рост памяти |
| `MemoryHigh=1536M` | Мягкий лимит — система начинает активно освобождать память | Замедляет процесс вместо его завершения |
| `MemoryMax=2G` | Жёсткий лимит — процесс будет перезапущен при превышении | Защищает сервер от зависания |
| `Restart=always` | Автоматический перезапуск при любом сбое | Gateway восстановится без вашего вмешательства |

#### Шаг 6.6. Защита директории с секретами

Директория `~/.openclaw/` хранит OAuth-токены (токены авторизации подключённых сервисов), сессионные ключи и другие секреты в открытом виде. Если другой пользователь на сервере получит к ним доступ, он сможет управлять вашим ассистентом.

```bash
# Protect the secrets directory
chmod 700 ~/.openclaw/
```

Права доступа `700` означают: только владелец может читать, писать и входить в эту директорию. Все остальные пользователи системы не увидят даже список файлов внутри — как запертая комната, ключ от которой есть только у вас.

#### Шаг 6.7. Настройка изоляции сессий

По умолчанию OpenClaw объединяет контекст всех пользователей, пишущих боту, в единую сессию (dmScope — direct message scope). Это означает, что секреты или файлы, загруженные одним пользователем, могут быть прочитаны другим. Даже если вы единственный пользователь, рекомендуется включить изоляцию:

```bash
openclaw config set session.dmScope per-channel-peer
```

Значение `per-channel-peer` создаёт отдельную сессию для каждой комбинации «канал + собеседник». Ваши сообщения в Telegram не смешиваются с сообщениями в WhatsApp, а если кто-то другой напишет вашему боту — он не увидит вашу переписку.

#### Шаг 6.8. Ограничение инструментов в групповых чатах

Если ваш бот добавлен в групповой чат, по умолчанию все участники группы могут использовать его инструменты — включая выполнение команд и доступ к файлам. Это опасно.

Добавьте ограничения для групп в конфигурацию. Откройте `~/.openclaw/openclaw.json5` и добавьте в блок `channels`:

```json5
"channels": {
  "telegram": {
    "linkPreview": false,
    "groups": {
      "*": {
        "requireMention": true,
        "tools": {
          "allow": ["read", "message"],
          "deny": ["exec", "write", "edit", "browser", "gateway", "nodes"]
        }
      }
    }
  }
}
```

Что это делает:
- `"*"` — правило распространяется на **все** группы
- `requireMention: true` — бот отвечает только при упоминании (@bot_name), а не на каждое сообщение
- `deny` — запрещает выполнение команд (`exec`), запись файлов (`write`, `edit`), браузер (`browser`) и управление шлюзом (`gateway`, `nodes`)
- Остаются только чтение (`read`) и отправка сообщений (`message`)

#### Шаг 7. Генерация токена авторизации

Токен — это длинная случайная строка, которая служит паролем для доступа к Gateway. В отличие от обычного пароля, его невозможно подобрать.

```bash
openssl rand -hex 32
```

Ожидаемый вывод — строка из 64 символов (буквы и цифры):
```
a1b2c3d4e5f6...  (64 characters)
```

Скопируйте эту строку и вставьте в конфигурацию вместо `INSERT_TOKEN_FROM_STEP_7_HERE`.

Сохраните файл конфигурации: `Ctrl+O`, `Enter`, `Ctrl+X`.

Важно: сохраните токен в надёжном месте (менеджер паролей). Если потеряете — придётся генерировать новый.

#### Шаг 7.1. Защита токена авторизации

OpenClaw может передавать токен через URL-параметры (`?token=...`), что ведёт к утечке через историю браузера, заголовки Referer и логи прокси.

**Правила защиты токена:**

1. **Длина токена** — используйте только `openssl rand -hex 32` (256 бит). OpenClaw не проверяет длину токена — слабый токен вроде `1234` будет принят, но это критическая уязвимость
2. **Не отключайте аутентификацию устройств** — настройка `dangerouslyDisableDeviceAuth: false` добавляет второй уровень защиты через одноразовые коды сопряжения (кратковременный код, который Gateway отображает в терминале при первом подключении нового устройства)
3. **Очистите историю браузера** после первого входа в Control UI
4. **Регулярно меняйте токен** — раз в месяц или при подозрении на утечку:

```bash
openclaw config set gateway.auth.token $(openssl rand -hex 32)
```

5. **Никогда не сохраняйте URL с токеном** в закладках и не передавайте через мессенджеры

#### Шаг 8. Постановка конфигурации под контроль версий

Контроль версий — это система, которая запоминает каждое изменение в файлах. Если кто-то (или что-то) изменит вашу конфигурацию, вы сможете увидеть, что именно изменилось, и откатить изменения.

```bash
cd ~/.openclaw && git init && git add . && git commit -m "baseline"
```

Эта команда:
1. Переходит в директорию настроек OpenClaw
2. Инициализирует Git-репозиторий (систему контроля версий)
3. Добавляет все файлы
4. Создаёт первый «снимок» — точку отсчёта

Ожидаемый вывод заканчивается:
```
[main (root-commit) xxxxxxx] baseline
 X files changed, Y insertions(+)
```

В дальнейшем можно проверить, изменилась ли конфигурация:

```bash
cd ~/.openclaw && git diff
```

Если вывод пустой — ничего не изменилось. Если есть изменения — вы увидите, что именно было изменено.

### Типичные проблемы

**Проблема: Docker не запускается**

Проверьте статус сервиса:
```bash
sudo systemctl status docker
```

Если статус не `active (running)`:
```bash
sudo systemctl start docker
sudo systemctl enable docker
```

Если Docker выдаёт ошибку `permission denied`:
```bash
sudo usermod -aG docker $USER
```

После этого **необходимо** выйти и заново зайти на сервер.

**Проблема: Неправильная версия Node.js**

```bash
node -v
```

Если версия не начинается с `v22`:
```bash
curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash -
sudo apt install -y nodejs
```

**Проблема: команда `openclaw` не найдена после установки**

Если терминал пишет `command not found: openclaw`:
```bash
export PATH="$PATH:$(pnpm bin -g)"
```

Чтобы изменение сохранилось после повторного входа:
```bash
echo 'export PATH="$PATH:$(pnpm bin -g)"' >> ~/.bashrc
source ~/.bashrc
```

**Проблема: Permission denied при запуске OpenClaw**

Убедитесь, что вы работаете под обычным пользователем, а не root. OpenClaw намеренно не запускается от root в целях безопасности:
```bash
whoami
```

Вывод должен содержать ваше имя пользователя, **не** `root`. Если вы root — выйдите и зайдите как обычный пользователь.

**Проблема: `openclaw security audit` показывает предупреждения**

Попробуйте автоматическое исправление безопасных проблем:
```bash
openclaw security audit --fix
```

Команда исправит то, что можно исправить автоматически, и покажет оставшиеся проблемы, которые нужно решить вручную.

### Что мы делаем и зачем

В предыдущих фазах мы подготовили сервер: установили OpenClaw, настроили безопасность и организовали VPN-доступ. Но система пока молчит — к ней не подключён ни мессенджер, ни AI-модель.

В этой фазе мы получим первый рабочий результат: отправим сообщение боту в Telegram и получим ответ от AI. Для этого нужно сделать три вещи: получить ключ доступа к AI-сервису, создать Telegram-бота и связать всё это внутри OpenClaw.

### Диаграмма

```
                        Internet
                           │
You (Telegram) ───► Telegram Bot API ───► OpenClaw Gateway ───► Claude/GPT API
                                          (your server)          (AI service)
      ◄── response ◄──────────────── ◄── response ◄───────── ◄── response ◄──
```

### Шаги

**Шаг 1. Получить API-ключ от AI-сервиса**

API-ключ (Application Programming Interface key) — это пароль для доступа к AI-сервису, как пароль от Wi-Fi: без него ваш сервер не сможет отправлять запросы к Claude или GPT.

Выберите один из вариантов:

*Вариант A — Claude (Anthropic):*

1. Перейдите на https://console.anthropic.com
2. Зарегистрируйтесь и подтвердите email
3. Перейдите в раздел "API Keys"
4. Привяжите способ оплаты (карту) — AI-сервисы взимают плату в зависимости от объёма обработанного текста; стоимость типичного диалога — несколько центов
5. Нажмите "Create Key", дайте ключу понятное имя, например `openclaw-vps`
6. Скопируйте ключ — он показывается только один раз

*Вариант B — GPT (OpenAI):*

1. Перейдите на https://platform.openai.com
2. Зарегистрируйтесь и подтвердите email
3. Перейдите в раздел "API keys"
4. Привяжите способ оплаты
5. Нажмите "Create new secret key", дайте ему имя `openclaw-vps`
6. Скопируйте ключ

Рекомендация по безопасности: установите лимит расходов в настройках аккаунта провайдера. Начните с $5-10 в месяц — для личного использования этого достаточно.

ВАЖНО: никогда не отправляйте API-ключ в чатах, не храните его в заметках или документах с общим доступом. Относитесь к нему как к паролю от банковского приложения.

---

**Шаг 2. Создать Telegram-бота**

Telegram-бот — это специальный аккаунт в Telegram, которым управляет программа (в нашем случае — OpenClaw), а не человек. Вы будете отправлять ему сообщения, а AI будет отвечать.

Бот создаётся через специального бота `@BotFather` — это официальный инструмент Telegram для управления ботами.

1. Откройте Telegram и найдите `@BotFather` (с синей галочкой верификации)
2. Отправьте ему команду:

```
/newbot
```

3. BotFather спросит имя бота — это отображаемое имя, например `My AI Assistant`
4. Затем спросит username — это техническое имя, должно заканчиваться на `bot`, например `my_ai_helper_bot`
5. BotFather ответит сообщением, содержащим токен бота — строку вида `7123456789:AAF...`

Токен бота — это адрес и пароль вашего бота в одной строке. Кто владеет токеном — тот управляет ботом. Храните его так же надёжно, как API-ключ.

---

**Шаг 3. Подключить канал Telegram к OpenClaw**

Подключитесь к серверу по SSH (как делали в предыдущих фазах) и выполните:

```bash
openclaw channels add --channel telegram --token 'YOUR_TELEGRAM_BOT_TOKEN'
```

Замените `YOUR_TELEGRAM_BOT_TOKEN` на реальный токен, полученный от BotFather (строка вида `7123456789:AAF...`).

Далее настройте allowlist (список тех, кому разрешено общаться с ботом). Это важно: без allowlist любой пользователь Telegram может написать вашему боту и тратить ваш API-бюджет.

Проверьте, что канал добавлен:

```bash
openclaw channels list
```

Ожидаемый результат:

```
Channel    Status     Bot Username
telegram   configured @my_ai_helper_bot
```

**Шаг 3.5. Отключить превью ссылок**

```bash
# Отключить превью ссылок — защита от утечки данных
openclaw config set channels.telegram.linkPreview false
```

Зачем это нужно: вредоносная веб-страница может заставить агента сгенерировать URL, содержащий секретные данные (например, токены или фрагменты вашей переписки). Мессенджер автоматически загрузит превью этого URL, отправив данные атакующему без вашего ведома. Отключение превью блокирует этот вектор атаки — ссылки по-прежнему будут кликабельными, но их содержимое не будет загружаться автоматически.

---

**Шаг 4. Настроить AI-модель и безопасность агента**

Укажите, какую AI-модель использовать. Для Claude Sonnet (рекомендуется как баланс качества и стоимости):

```bash
openclaw config set agents.defaults.model anthropic/claude-sonnet-4-6
```

Сохраните API-ключ в конфигурации окружения:

```bash
openclaw config set env.ANTHROPIC_API_KEY "your-api-key"
```

Замените `your-api-key` на реальный ключ, полученный на Шаге 1. Кавычки обязательны.

Если вы выбрали OpenAI, используйте вместо этого:

```bash
openclaw config set agents.defaults.model openai/gpt-4o
openclaw config set env.OPENAI_API_KEY "your-api-key"
```

Включите полную изоляцию — все инструменты агента будут работать внутри Docker-контейнера, без доступа к основной системе:

```bash
openclaw config set agents.defaults.sandbox.mode all
```

Установите минимальный набор инструментов — агент сможет только вести диалог, без доступа к файлам, браузеру или командной строке:

```bash
openclaw config set agents.defaults.tools.profile minimal
```

---

**Шаг 5. Запустить Gateway**

Gateway — центральный процесс OpenClaw, который принимает сообщения из Telegram, передаёт их AI и возвращает ответы. Он должен работать постоянно.

```bash
openclaw gateway restart
```

Ожидаемый результат:

```
Gateway restarted successfully
Listening on 127.0.0.1:18789
Channels: telegram (connected)
```

**Шаг 6. Отправить первое сообщение**

Откройте Telegram, найдите вашего бота по username (например, `@my_ai_helper_bot`) и отправьте:

```
Hello! Tell me about yourself.
```

В течение нескольких секунд бот должен ответить сообщением от AI.

### Проверка

1. Отправьте боту сообщение «Привет!» в Telegram — ответ должен прийти в течение 5-15 секунд

2. Проверьте статус системы:

```bash
openclaw status
```

Ожидаемый результат — активная сессия, подключённый канал Telegram, название модели.

3. Проверьте логи на наличие ошибок:

```bash
openclaw logs
```

Ожидаемый результат — записи о полученном сообщении, отправленном запросе к AI и доставленном ответе, без строк, содержащих `ERROR`.

### Частые проблемы

**«Бот не отвечает, логи пустые»**

Токен бота введён неправильно или Gateway не запущен. Проверьте токен: `openclaw channels status`. Перезапустите: `openclaw gateway restart`.

**«Unauthorized» или «401» в логах**

API-ключ неправильный, просрочен или не активирован. Проверьте ключ в панели провайдера (console.anthropic.com или platform.openai.com). Убедитесь, что привязан способ оплаты.

**«Ответ приходит 30+ секунд»**

Нормально для первого сообщения — AI-сервис «прогревается». Также зависит от загрузки серверов провайдера. Подождите. Последующие сообщения будут быстрее.

**«Rate limited» или «429» в логах**

Вы отправляете слишком много сообщений за короткое время. У каждого провайдера есть лимит запросов в минуту. Подождите минуту и попробуйте снова. Для новых аккаунтов лимиты ниже.

**«Insufficient funds» или «402»**

Баланс API-аккаунта исчерпан. Пополните баланс в панели провайдера.

**«Бот отвечает, но не мне»**

Allowlist настроен неправильно, и кто-то другой нашёл вашего бота. Пересмотрите allowlist: `openclaw channels add --channel telegram --token 'YOUR_TOKEN'` (перенастройте).

## Фаза 5: Команда агентов

### Что мы делаем и зачем

В Фазе 4 мы запустили одного универсального агента. Он может вести разговор, но не умеет писать код, работать с файлами или искать в интернете. Давать одному агенту все возможности сразу — плохая идея с точки зрения безопасности: если AI ошибётся или будет обманут специально составленным сообщением (prompt injection — атака через текст, заставляющая AI выполнять нежелательные действия), он будет иметь доступ ко всему сразу.

Представьте, что у вас есть команда помощников. Каждый — специалист в своей области. Один отвечает на вопросы, другой пишет код, третий ищет информацию в интернете. Все они работают на одном сервере, но изолированы друг от друга — как сотрудники в отдельных кабинетах с разными ключами доступа.

В этой фазе мы настроим трёх агентов:

| Агент | Назначение | Что может | Чего НЕ может |
|-------|-----------|-----------|---------------|
| **Assistant** | Повседневные вопросы, разговор | Только разговор | Файлы, код, интернет |
| **Coder** | Код, файлы, технические задачи | Чтение/запись файлов, выполнение команд | Интернет, браузер |
| **Researcher** | Поиск информации в интернете | Браузер (в изоляции) | Файлы, выполнение команд |

### Диаграмма

```
Telegram ──► OpenClaw Gateway
               │
               │  message ──► routing by command
               │
               ├── (default) ──────► Assistant (Claude, conversation only)
               │                        └── Sandbox A (isolated, no tools)
               │
               ├── /coder ──────────► Coder (GPT-4o, files + code)
               │                        └── Sandbox B (isolated, access to /workspaces/coder)
               │
               └── /research ───────► Researcher (Claude, browser)
                                        └── Sandbox C (isolated, no file access)

Each agent = separate Docker container + own API key + own permissions
```

### Шаги

**Шаг 1. Создать рабочую директорию для Coder**

Рабочая директория — это папка на сервере, куда агент-кодер может сохранять файлы. Агент будет видеть только эту папку и ничего за её пределами.

```bash
sudo mkdir -p /opt/openclaw/workspaces/coder
sudo chown openclaw:openclaw /opt/openclaw/workspaces/coder
```

Ожидаемый результат: команды выполняются без ошибок. Проверьте:

```bash
ls -la /opt/openclaw/workspaces/
```

Ожидаемый результат:

```
drwxr-xr-x 2 openclaw openclaw 4096 ... coder
```

---

**Шаг 2. Получить дополнительные API-ключи (при необходимости)**

Если вы хотите использовать разных AI-провайдеров для разных агентов (например, Claude для ассистента и GPT-4o для кодера), вам понадобятся API-ключи от обоих сервисов. Создайте их по инструкции из Фазы 4, Шаг 1.

Если вы хотите использовать одного провайдера для всех агентов — достаточно одного ключа. Тем не менее рекомендуется создавать отдельные ключи для каждого агента: это позволяет отслеживать расходы по агентам отдельно и отключать один ключ, не затрагивая остальные.

---

**Шаг 3. Сохранить API-ключи в переменных окружения**

Переменные окружения — это именованные значения, доступные программам на сервере. Они безопаснее, чем запись ключей непосредственно в конфигурационный файл, потому что не попадут в историю команд или резервные копии конфигурации.

Откройте файл окружения для OpenClaw:

```bash
sudo nano /etc/openclaw/env
```

Добавьте строки с вашими ключами (замените значения на реальные):

```
AGENT_ASSISTANT_ANTHROPIC_KEY=sk-ant-your-key-for-assistant
AGENT_CODER_OPENAI_KEY=sk-your-key-for-coder
AGENT_RESEARCHER_ANTHROPIC_KEY=sk-ant-your-key-for-researcher
```

Сохраните файл (Ctrl+O, Enter, Ctrl+X) и ограничьте доступ к нему:

```bash
sudo chmod 600 /etc/openclaw/env
sudo chown openclaw:openclaw /etc/openclaw/env
```

`chmod 600` означает: только владелец файла может его читать и редактировать. Никто другой на сервере не получит доступ к ключам.

Чтобы OpenClaw загружал эти переменные при запуске, укажите путь к файлу в конфигурации systemd:

```bash
sudo mkdir -p /etc/systemd/system/openclaw.service.d
sudo tee /etc/systemd/system/openclaw.service.d/env.conf << 'EOF'
[Service]
EnvironmentFile=/etc/openclaw/env
EOF
sudo systemctl daemon-reload
```

Команда `daemon-reload` заставляет systemd перечитать конфигурации сервисов. После этого при каждом запуске OpenClaw переменные из файла `/etc/openclaw/env` будут автоматически загружаться.

---

#### Шаг 2.5. Правила безопасности для учётных данных

Подключение OpenClaw к внешним сервисам (GitHub, Slack, Google Workspace, календарь) создаёт так называемый «Shadow AI» — неконтролируемого агента с унаследованными правами пользователя. Если агент будет скомпрометирован через prompt injection, атакующий наследует **все** права подключённых сервисов.

**Правила:**

1. **Только выделенные сервисные аккаунты** — никогда не используйте личные или рабочие аккаунты для предоставления доступа агентам
2. **Минимальные права** — GitHub: токены с доступом только к конкретным репозиториям, только чтение. Slack: только `channels:read` + `chat:write`, никогда не админские права. Google: `gmail.readonly`, а не `mail.google.com`
3. **Короткоживущие токены** — по возможности используйте токены с временем жизни 5 минут (снижает риск кражи на 92% по данным Okta 2025)
4. **Любые учётные данные, попавшие в контекст AI-модели, следует считать скомпрометированными** — немедленно ротируйте их

---

**Шаг 4. Создать конфигурацию с тремя агентами**

Создайте конфигурационный файл Gateway — это основной файл, описывающий всех агентов, их права и маршрутизацию сообщений.

> **Важно:** В Фазе 3 мы создали файл `openclaw.json5` с общими настройками безопасности (привязка к loopback, токен авторизации, режим sandbox). Файл `gateway.yaml` дополняет его — здесь мы описываем конкретных агентов, их модели и маршрутизацию. Оба файла находятся в `~/.openclaw/` и работают совместно: `openclaw.json5` задаёт глобальные правила безопасности, а `gateway.yaml` — индивидуальные настройки для каждого агента.

```bash
nano ~/.openclaw/gateway.yaml
```

Вставьте следующее содержимое целиком:

```yaml
# Основной конфигурационный файл OpenClaw Gateway
# Описывает всех агентов, их права доступа и маршрутизацию сообщений

gateway:
  bind: loopback     # Gateway слушает только на localhost (127.0.0.1)
  port: 18789
  auth:
    mode: token      # Доступ к Gateway только по токену

# Настройки агентов
agents:

  # Настройки по умолчанию для всех агентов
  defaults:
    sandbox:
      mode: all          # Все инструменты работают внутри Docker
      scope: dedicated   # Каждый агент — отдельный контейнер
      docker: true
    tools:
      profile: minimal   # По умолчанию — минимальный набор инструментов
    workspaceAccess: none # По умолчанию — нет доступа к файлам

  # Определение каждого агента
  definitions:

    # Assistant — повседневные вопросы, разговор
    # Без инструментов, без доступа к файлам
    assistant:
      model:
        provider: anthropic
        name: claude-sonnet-4-6
      tools:
        profile: minimal
      sandbox:
        mode: all
        scope: dedicated
        workspaceAccess: none
      credentials:
        anthropicApiKey: ${AGENT_ASSISTANT_ANTHROPIC_KEY}

    # Coder — генерация кода, работа с файлами
    # Может читать и записывать файлы, выполнять команды
    # НО только в своей рабочей директории, без доступа в интернет
    coder:
      model:
        provider: openai
        name: gpt-4o
      tools:
        profile: coding
        allow: [file_read, file_write, shell_exec]
        deny: [browser, network_fetch]
      sandbox:
        mode: all
        scope: dedicated
        workspaceAccess: rw
        workspaceDir: /opt/openclaw/workspaces/coder
      credentials:
        openaiApiKey: ${AGENT_CODER_OPENAI_KEY}

    # Researcher — поиск информации в интернете
    # Может открывать веб-страницы в изолированном браузере
    # НО не может записывать файлы или выполнять команды
    researcher:
      model:
        provider: anthropic
        name: claude-sonnet-4-6
      tools:
        profile: minimal
        allow: [browser]
        deny: [file_write, shell_exec]
      sandbox:
        mode: all
        scope: dedicated
        workspaceAccess: none
      credentials:
        anthropicApiKey: ${AGENT_RESEARCHER_ANTHROPIC_KEY}

# Каналы связи
channels:
  telegram:
    defaultAgent: assistant    # Обычные сообщения идут к Assistant
    agentRouting:              # Команды для переключения на других агентов
      /coder: coder
      /research: researcher
```

Сохраните файл (Ctrl+O, Enter, Ctrl+X).

**Что здесь происходит:**

- `defaults` задаёт правила безопасности, которые применяются ко всем агентам, если не переопределены для конкретного агента
- Каждый агент в `definitions` описывает: какую AI-модель использовать, какие инструменты разрешены/запрещены, какой уровень доступа к файлам
- `${AGENT_...}` — ссылки на переменные окружения из Шага 3. Gateway подставит реальные ключи при запуске
- `agentRouting` определяет, как переключаться между агентами в Telegram с помощью команд

**Почему безопасность настроена именно так:**

| Агент | Инструменты | Файлы | Интернет | Обоснование |
|-------|------------|-------|----------|-------------|
| Assistant | Нет | Нет | Нет | Минимальная поверхность атаки. Если AI обманут через сообщение — он ничего не сможет, кроме как ответить текстом |
| Coder | Файлы, команды | Только `/workspaces/coder` | Нет | Может работать с кодом, но только в отведённой папке. Не имеет доступа к интернету — значит, не может отправить ваши данные наружу |
| Researcher | Браузер | Нет | Да (через sandbox) | Может искать информацию, но не может сохранять файлы на сервер. Браузер работает внутри Docker-контейнера, изолированно от основной системы |

> **Безопасность при чтении:** Агент, работающий с внешним контентом (веб-страницы, документы), наиболее уязвим для атак типа «indirect prompt injection» (когда вредоносные инструкции скрыты в содержимом веб-страницы или документа, и AI выполняет их, думая, что это часть задачи). Поэтому агенту-исследователю запрещён доступ к файлам памяти и конфигурации — даже если атакующий «обманет» его через подготовленную страницу, агент не сможет изменить системные настройки или данные других агентов.

---

**Шаг 5. Защитить файлы идентичности агентов**

Файлы `SOUL.md` и `MEMORY.md` определяют «личность» и «память» ваших агентов — инструкции, по которым AI ведёт себя определённым образом. Атака «memory poisoning» — это когда вредоносный контент модифицирует эти файлы, создавая постоянный бэкдор (скрытый вход), который переживает перезагрузки и работает незаметно.

Защитите эти файлы от модификации:

```bash
# Защита файлов идентичности агентов от отравления
sudo chattr +i ~/openclaw/workspace/SOUL.md
sudo chattr +i ~/openclaw/workspace/MEMORY.md
```

Теперь, чтобы обновить файлы идентичности, нужно явно снять защиту (`sudo chattr -i`), внести изменения и восстановить защиту — точно так же, как мы делали с конфигурацией SSH в Фазе 2.

---

**Шаг 5.5. Ограничить ресурсы контейнеров агентов**

Без ограничений один «зависший» контейнер агента может потребить всю оперативную память сервера и обрушить систему — включая Gateway и остальных агентов. Это как если бы один жилец в многоквартирном доме открыл все краны на полную — воды не хватит остальным.

В файле `gateway.yaml` обновите секцию `defaults` в блоке `sandbox`, добавив ограничения ресурсов:

```yaml
sandbox:
  mode: all
  scope: dedicated
  docker: true
  memory: "512m"
  cpus: "0.5"
```

Параметр `memory: "512m"` ограничивает каждый контейнер агента 512 МБ памяти, а `cpus: "0.5"` — половиной одного ядра CPU.

Расчёт для VPS с 4 ГБ RAM: Gateway (1,5 ГБ) + 3 агента по 512 МБ (1,5 ГБ) + система (0,5 ГБ) = ~3,5 ГБ — укладывается в лимит с запасом.

### Частые проблемы

| Симптом | Причина | Решение |
|---------|---------|---------|
| Агент не переключается по команде `/coder` или `/research` | Маршрутизация не настроена или Gateway не перечитал конфигурацию | Проверьте секцию `agentRouting` в `gateway.yaml`. Перезапустите: `openclaw gateway restart` |
| Coder не может записывать файлы, ошибка `Permission denied` | Директория `/opt/openclaw/workspaces/coder` принадлежит другому пользователю | Исправьте владельца: `sudo chown openclaw:openclaw /opt/openclaw/workspaces/coder` |
| Researcher не открывает веб-страницы | Браузер не разрешён в конфигурации sandbox или Docker не может запустить headless-браузер (браузер без графического интерфейса, работающий в фоне) | Проверьте `allow: [browser]` в конфигурации. Пересоздайте sandbox: `openclaw sandbox recreate` |
| На вопрос отвечает не тот агент | Неправильно указан `defaultAgent` или запуталась маршрутизация | Проверьте `defaultAgent: assistant` и секцию `agentRouting` в `gateway.yaml` |
| `Error: credential not found` при запуске | Переменные окружения не загружены | Проверьте файл `/etc/openclaw/env` и перезапустите сервис: `openclaw gateway restart` |
| Один из контейнеров не запускается | Docker не хватает ресурсов (памяти или места на диске) | Проверьте: `docker ps -a` для статуса контейнеров, `df -h` для свободного места, `free -m` для памяти |
| `security audit` показывает FAIL | В конфигурации есть небезопасные параметры | Запустите `openclaw security audit --fix` для автоматического исправления безопасных проблем, остальное исправьте вручную по описаниям |

## Фаза 6: Обслуживание и мониторинг

### Что мы делаем и зачем

В предыдущих фазах мы установили и настроили OpenClaw на VPS. Но сервер — это не бытовой прибор, который можно «настроить и забыть». Без регулярного обслуживания настройки безопасности могут «поплыть» (кто-то случайно меняет конфиг через чат), обновления не будут установлены, а расходы на API могут выйти из-под контроля.

В этой фазе мы настроим простую процедуру обслуживания: 5 минут в неделю и 15 минут в месяц — этого достаточно, чтобы система оставалась безопасной и стабильной.

> **Совет:** Если вы использовали автоматический установщик, опциональный скрипт `06-maintenance.sh` может автоматизировать часть этой процедуры обслуживания. Запустите `bash /opt/openclaw/scripts/06-maintenance.sh`, чтобы настроить автоматические еженедельные проверки и резервные копии.

### Схема

```
┌── Weekly (5 minutes) ────────────────────────────────┐
│                                                       │
│  security audit → git diff → cron list                │
│  → reboot check → token usage                        │
│                                                       │
├── Monthly (15 minutes) ──────────────────────────────┤
│                                                       │
│  openclaw update → apt upgrade                        │
│  → backup → spending review                          │
│                                                       │
├── As needed (on failures) ───────────────────────────┤
│                                                       │
│  logs → doctor → restart → config rollback            │
│                                                       │
└───────────────────────────────────────────────────────┘
```

#### 1.5. Ежедневный автоматический аудит (настраивается один раз)

Вместо ежедневной ручной проверки сервера мы настроим автоматический ночной аудит — скрипт, который проверяет ключевые показатели безопасности и записывает отчёт в лог-файл. Это как видеонаблюдение: запись идёт круглосуточно, а просматриваете вы её только если что-то случилось. Подход основан на рекомендациях по аудиту безопасности от SlowMist.

Создайте файл скрипта:

```bash
sudo mkdir -p /opt/security
sudo nano /opt/security/nightly-audit.sh
```

Вставьте содержимое:

```bash
#!/bin/bash
# /opt/security/nightly-audit.sh
# Nightly security audit

REPORT="/var/log/security-audit/$(date +%Y-%m-%d).log"
mkdir -p /var/log/security-audit

echo "=== Audit $(date) ===" > "$REPORT"

# 1. New open ports
ss -tlnp >> "$REPORT"

# 2. Changes in /etc
cd /opt/security/baselines 2>/dev/null && \
  find /etc -type f -exec sha256sum {} + 2>/dev/null | \
  diff etc_hashes.baseline - >> "$REPORT" 2>/dev/null

# 3. OpenClaw security check
openclaw security audit --deep >> "$REPORT" 2>/dev/null

# 4. Configuration check
cd ~/.openclaw && git diff >> "$REPORT"

# 5. Gateway memory usage
ps -o rss=,vsz=,pid=,comm= -C node >> "$REPORT"

echo "=== Completed ===" >> "$REPORT"
```

Установите скрипт и добавьте его в расписание cron (планировщик задач — программа, автоматически запускающая команды по расписанию, как будильник для скриптов):

```bash
sudo chmod 700 /opt/security/nightly-audit.sh
echo "0 3 * * * root /opt/security/nightly-audit.sh" | sudo tee /etc/cron.d/security-audit
```

Скрипт будет запускаться каждую ночь в 3:00 и записывать отчёт в `/var/log/security-audit/`. Просмотр отчётов:

```bash
cat /var/log/security-audit/$(date +%Y-%m-%d).log
```

#### 1.6. Безопасность навыков (Skills)

Навыки (Skills) — это дополнительные возможности, которые можно установить из каталога ClawHub (как приложения из магазина).

В январе-феврале 2026 года произошла масштабная атака на цепочку поставок ClawHub, получившая название **ClawHavoc** — более 800 вредоносных навыков (около 20% всего каталога) содержали трояны (программы, замаскированные под полезные), кейлоггеры (перехватчики нажатий клавиш) и стилеры (программы для кражи паролей и токенов). Многие из этих навыков были точными копиями популярных, с минимальными изменениями в названии.

Правила безопасности при работе с навыками:

1. Никогда не устанавливайте навыки без предварительного аудита исходного кода — просмотрите файлы навыка на GitHub перед установкой
2. Фиксируйте версии: `clawhub install <skill> --version X.Y.Z` — это гарантирует, что навык не обновится автоматически до вредоносной версии
3. Не выполняйте команды из раздела «Prerequisites» в SKILL.md — вредоносные навыки часто просят выполнить опасные команды под видом «подготовки к установке»
4. Используйте ClawVet для сканирования навыков перед установкой: `github.com/MohibShaikh/clawvet`

**Автоматическое сканирование навыков**

Ручной аудит полезен, но недостаточен в масштабах кампании ClawHavoc (800+ вредоносных навыков). Используйте автоматические сканеры:

| Инструмент | Что делает | Установка |
|------|-------------|-------------|
| Cisco AI Skill Scanner | Статический + семантический анализ + VirusTotal | `pip install cisco-ai-skill-scanner` |
| Aguara | 177 правил обнаружения, не требует API-ключей | github.com/garagon/aguara |

Пример сканирования перед установкой навыка:

```bash
skill-scanner scan /path/to/skill --policy strict
```

Если сканер обнаружил проблемы уровня `high` или `critical` — не устанавливайте навык.

#### 3. Контроль расходов

AI-модели тарифицируются по количеству токенов — чем больше текста вы отправляете и получаете, тем дороже. Без контроля расходов счета в конце месяца могут неприятно удивить.

**Шаг 3.1. Проверка расходов на стороне провайдера**

Данные `openclaw status` — это оценка. Для точных цифр проверяйте панели управления провайдеров:

- **Anthropic (Claude):** https://console.anthropic.com — раздел «Usage»
- **OpenAI (GPT):** https://platform.openai.com/usage — раздел «Usage»

Заходите туда хотя бы раз в месяц.

**Шаг 3.2. Установка лимитов расходов**

Обязательно установите лимиты расходов у каждого провайдера — это страховка от непредвиденных затрат:

- **Anthropic:** https://console.anthropic.com → Settings → Spending Limits
- **OpenAI:** https://platform.openai.com/settings → Billing → Usage limits

Рекомендуемые начальные лимиты для личного использования: $10-20 в месяц на провайдера. Скорректируйте позже на основе фактического потребления.

**Шаг 3.3. Зачем отдельные API-ключи для каждого агента**

В Фазе 4 мы настроили отдельный API-ключ для каждого агента. Это не только мера безопасности — это ещё и инструмент контроля расходов. Если у каждого агента свой ключ, вы можете видеть в панели управления провайдера, сколько именно тратит каждый агент. А если один агент выходит из-под контроля, можно отключить только его ключ, не затрагивая остальных.

**Шаг 3.4. Приблизительные ежемесячные расходы**

Ориентировочные расходы при типичном личном использовании (10-30 сообщений в день):

| Сценарий | Модель | Приблизительная стоимость в месяц |
|----------|-------|--------------------------|
| Лёгкое использование (10 сообщений/день, короткие диалоги) | Claude Sonnet | $3-7 |
| Среднее использование (20-30 сообщений/день, средние диалоги) | Claude Sonnet | $8-15 |
| Интенсивное использование (50+ сообщений/день, длинные диалоги с документами) | Claude Sonnet | $20-40 |
| Использование GPT-4o вместо Claude | GPT-4o | Примерно тот же диапазон |

Сверху добавляется стоимость VPS-сервера (около $5-10/мес). Общий типичный бюджет: **$15-25 в месяц** за полностью приватного персонального AI-ассистента.

#### 4. Резервное копирование

Если сервер выйдет из строя, вам потребуется восстановить конфигурацию. Всё важное хранится в одной директории: `~/.openclaw/`.

**Шаг 4.1. Что нужно копировать**

Директория `~/.openclaw/` содержит:
- `openclaw.json` — основной файл конфигурации
- Ключи и токены каналов (Telegram, WhatsApp)
- Настройки агентов
- Историю сессий

Эта директория уже находится под контролем версий git (мы настроили это в Фазе 3), так что у вас есть история всех изменений на самом сервере. Но если сервер исчезнет — исчезнет и эта история. Поэтому нужна копия на вашем компьютере.

**Шаг 4.2. Копирование на ваш компьютер**

Выполните эту команду **на вашем локальном компьютере** (не на сервере):

```bash
scp -r openclaw@your-server:~/.openclaw/ ~/openclaw-backup-$(date +%Y-%m-%d)/
```

Команда `scp` (Secure Copy) скачивает всю директорию `~/.openclaw/` с сервера в папку на вашем компьютере с датой в названии.

Ожидаемый результат:

```
openclaw.json                     100%  2.1KB   125.3KB/s   00:00
...
```

**Шаг 4.3. Простой скрипт для регулярного резервного копирования**

Чтобы не набирать команду вручную каждый раз, создайте файл скрипта на вашем компьютере (набор команд, который можно запустить одним действием):

```bash
#!/bin/bash
BACKUP_DIR="$HOME/openclaw-backups"
DATE=$(date +%Y-%m-%d)
mkdir -p "$BACKUP_DIR"
scp -r openclaw@your-server:~/.openclaw/ "$BACKUP_DIR/backup-$DATE/"
echo "Backup saved to $BACKUP_DIR/backup-$DATE/"

# Delete backups older than 90 days
find "$BACKUP_DIR" -maxdepth 1 -type d -name "backup-*" -mtime +90 -exec rm -rf {} +
```

Сохраните его как `~/openclaw-backup.sh` и сделайте исполняемым:

```bash
chmod +x ~/openclaw-backup.sh
```

Запускайте раз в месяц:

```bash
~/openclaw-backup.sh
```

Ожидаемый результат:

```
Backup saved to /home/yourname/openclaw-backups/backup-2026-03-15/
```

**Шаг 4.4. Автоматическое резервное копирование памяти агентов**

Помимо полной копии конфигурации, нужно отдельно сохранять «мозг» агентов — файлы памяти. Если память агента отравлена (через атаку memory poisoning), можно откатить её к чистой версии из резервной копии.

Создайте скрипт:

```bash
sudo nano /opt/security/agent-brain-backup.sh
```

Вставьте содержимое:

```bash
#!/bin/bash
# /opt/security/agent-brain-backup.sh
BACKUP_DIR="/opt/backups/openclaw-brain"
AGENT_DIR="$HOME/.openclaw"
mkdir -p "$BACKUP_DIR"

cd "$BACKUP_DIR"
[ -d .git ] || git init

cp -r "$AGENT_DIR/memory/" "$BACKUP_DIR/memory/" 2>/dev/null
cp "$AGENT_DIR/openclaw.json" "$BACKUP_DIR/" 2>/dev/null

git add -A
git diff --cached --quiet || git commit -m "backup $(date +%Y-%m-%d_%H%M)"
```

Установите скрипт и добавьте в расписание (каждый час):

```bash
sudo chmod 700 /opt/security/agent-brain-backup.sh
echo "0 * * * * openclaw /opt/security/agent-brain-backup.sh" | sudo tee /etc/cron.d/agent-backup
```

Если потребуется восстановить память агента (например, после обнаружения подозрительного поведения):

```bash
cd /opt/backups/openclaw-brain
git log --oneline -10
git checkout <commit> -- memory/
cp -r memory/ ~/.openclaw/memory/
openclaw gateway restart
```

Замените `<commit>` на хеш нужного коммита из списка — например, последний «чистый» снимок до того, как агент начал вести себя странно.

#### 5. Обновление сервера

Операционная система сервера тоже нуждается в обновлениях. Автоматические обновления безопасности устанавливаются в Фазе 2 (пакет `unattended-upgrades`), но полное обновление всех пакетов лучше делать вручную раз в месяц.

**Шаг 5.1. Обновление списка пакетов и установка обновлений**

```bash
sudo apt update && sudo apt upgrade -y
```

Первая часть (`apt update`) скачивает свежий список доступных пакетов. Вторая (`apt upgrade -y`) устанавливает все обновления. Флаг `-y` означает автоматическое согласие на установку.

Ожидаемый результат:

```
Reading package lists... Done
Building dependency tree... Done
The following packages will be upgraded:
  libssl3 openssl ...
3 upgraded, 0 newly installed, 0 to remove.
...
```

**Шаг 5.2. Проверка необходимости перезагрузки**

```bash
cat /var/run/reboot-required 2>/dev/null || echo "No reboot needed"
```

Если перезагрузка необходима (обычно после обновления ядра), переходите к следующему шагу.

**Шаг 5.3. Безопасная перезагрузка сервера**

Перед перезагрузкой убедитесь, что OpenClaw установлен как системная служба (daemon — фоновый процесс, автоматически запускающийся при загрузке системы). Мы сделали это в Фазе 3 командой `openclaw service install`. В этом случае после перезагрузки OpenClaw запустится сам.

Проверьте, что служба настроена на автозапуск:

```bash
systemctl is-enabled openclaw
```

Ожидаемый результат:

```
enabled
```

Если вывод `enabled` — можно безопасно перезагружать:

```bash
sudo reboot
```

Сервер будет недоступен 1-2 минуты. После перезагрузки подключитесь снова и проверьте:

```bash
ssh openclaw@your-server
systemctl status openclaw
```

Ожидаемый результат:

```
● openclaw.service - OpenClaw Gateway
     Loaded: loaded (/etc/systemd/system/openclaw.service; enabled)
     Active: active (running) since ...
```

Статус `active (running)` означает, что шлюз запустился автоматически после перезагрузки. Ваши каналы (Telegram, WhatsApp) переподключатся самостоятельно в течение нескольких секунд.

#### 6. Когда что-то пошло не так

Если OpenClaw перестал отвечать на сообщения, работает с ошибками или ведёт себя странно — вот порядок диагностики, от простого к сложному.

**Шаг 6.1. Проверьте статус службы**

```bash
systemctl status openclaw
```

Если статус `active (running)` — служба работает, проблема в другом. Если `inactive` или `failed` — служба упала.

**Шаг 6.2. Прочитайте логи**

```bash
openclaw logs
```

Логи (журнал событий — записи всех действий системы) покажут последние события. Ищите строки с `ERROR` или `WARN` — они указывают на проблемы.

Чтобы увидеть только ошибки:

```bash
openclaw logs | grep -i error
```

Распространённые ошибки и их значение:

| Ошибка в логах | Что означает | Что делать |
|---------------|--------------|-----------|
| `ANTHROPIC_API_KEY invalid` | Недействительный или отозванный API-ключ | Обновите ключ в конфигурации |
| `rate limit exceeded` | Превышен лимит запросов к провайдеру | Подождите или увеличьте лимит у провайдера |
| `channel disconnected` | Потеряно соединение с Telegram/WhatsApp | Перезапустите шлюз (шаг 6.4) |
| `sandbox: docker not running` | Docker не запущен | `sudo systemctl start docker` |
| `ENOSPC` | Закончилось место на диске | Освободите место: `sudo apt autoremove && sudo docker system prune` |

**Шаг 6.3. Автоматическая диагностика**

```bash
openclaw doctor
```

Команда проверяет все компоненты системы: версию Node.js, Docker, состояние шлюза, подключения каналов, работу песочницы. Если что-то не так — покажет проблему и предложит решение.

**Шаг 6.4. Перезапуск шлюза**

Если проблема не решена — перезапустите шлюз:

```bash
openclaw gateway restart
```

Ожидаемый результат:

```
Stopping gateway... done
Starting gateway... done
Gateway is running
```

Перезапуск занимает несколько секунд. Каналы переподключатся автоматически.

Если `openclaw gateway restart` не помогает, попробуйте через systemd:

```bash
sudo systemctl restart openclaw
```

**Шаг 6.5. Проверьте статус службы**

```bash
systemctl status openclaw
```

Убедитесь, что статус `active (running)` и в последних строках вывода нет ошибок.

**Шаг 6.6. Откат конфигурации (крайняя мера)**

Если проблема появилась после изменения настроек и вы не можете разобраться, что именно сломалось — верните конфигурацию к последнему рабочему состоянию:

```bash
cd ~/.openclaw && git checkout .
```

Эта команда отменит все изменения конфигурации, сделанные после последнего `git commit`. Затем перезапустите шлюз:

```bash
openclaw gateway restart
```

Если нужно откатиться к конкретному сохранению — просмотрите историю:

```bash
cd ~/.openclaw && git log --oneline
```

Ожидаемый результат:

```
abc1234 before update 2026-03-15
def5678 before update 2026-02-15
789abcd baseline hardened config
```

Откатитесь к нужной точке:

```bash
cd ~/.openclaw && git checkout <commit-hash> -- .
openclaw gateway restart
```

Замените `<commit-hash>` на нужный идентификатор из списка (например, `789abcd`).

### Распространённые проблемы

**Проблема:** `openclaw security audit` показывает предупреждения, которых раньше не было.

Причина: конфигурация изменилась — возможно, после обновления OpenClaw добавились новые проверки, или кто-то изменил настройки.

Решение:
```bash
openclaw security audit --fix
cd ~/.openclaw && git diff
```
Просмотрите, что изменилось, и если изменения корректны — сохраните:
```bash
cd ~/.openclaw && git add . && git commit -m "post-audit fix $(date +%Y-%m-%d)"
```

**Проблема:** `git diff` показывает изменения, которые вы не вносили.

Причина: агент или обновление изменили конфигурацию.

Решение: внимательно прочитайте, что именно изменилось. Если изменения нежелательные:
```bash
cd ~/.openclaw && git checkout .
openclaw gateway restart
```

**Проблема:** `openclaw update` завершается с ошибкой.

Причина: проблема с сетью или конфликт зависимостей.

Решение:
```bash
openclaw doctor
sudo apt update && sudo apt upgrade -y
openclaw update
```
Если не помогло — проверьте, достаточно ли места на диске:
```bash
df -h /
```
Должно быть не менее 2 ГБ свободного места.

**Проблема:** каналы (Telegram/WhatsApp) отключились после перезагрузки.

Причина: обычно каналы переподключаются автоматически в течение 10-30 секунд. Если прошло больше минуты — возможно, истекла сессия.

Решение:
```bash
openclaw channels status
```
Если канал показывает статус `disconnected`:
```bash
openclaw channels login <channel-name>
```
Для WhatsApp потребуется заново отсканировать QR-код.

---

**Проблема:** использование токенов аномально высокое.

Причина: зацикленный агент, спам в канале или слишком длинные контексты диалогов.

Решение:
```bash
openclaw logs | grep -i "tokens"
openclaw status --all
```

```bash
openclaw channels remove <channel-name>
```

## Что дальше?

Поздравляем — у вас есть полностью настроенный и защищённый персональный AI-ассистент, работающий на вашем собственном сервере. Ваши данные под вашим контролем, система защищена пятью уровнями безопасности, а обслуживание занимает 5 минут в неделю.

Вот что можно сделать дальше, когда вы освоитесь с базовой настройкой:

**Добавить канал WhatsApp**

Если вы настроили только Telegram, добавьте WhatsApp — один шлюз может обслуживать несколько мессенджеров одновременно:

```bash
openclaw channels add whatsapp
```

Команда проведёт вас через процесс привязки — нужно будет отсканировать QR-код телефоном.

**Настроить автоматические задачи**

Запланируйте повторяющиеся задачи, которые агент будет выполнять самостоятельно. Например, ежедневную сводку новостей в 9 утра:

```bash
openclaw cron add --schedule "0 9 * * *" --agent assistant --prompt "Prepare a brief summary of yesterday's top news"
```

Другие идеи: напоминания, отчёты о расходах, мониторинг изменений на веб-страницах.

**Попробовать разные AI-модели**

OpenClaw поддерживает модели от разных провайдеров. Попробуйте создать второго агента с другой моделью и сравнить качество ответов:

- Claude Sonnet — хороший баланс скорости и качества
- Claude Opus — максимальное качество для сложных задач
- GPT-4o — альтернатива от OpenAI
- Модели с открытым кодом через совместимые API

**Изучить дополнительные роли агентов**

В Фазе 4 мы настроили базовых агентов. По мере необходимости можно добавить:

- **scheduler** — управление расписанием и напоминания
- **notifier** — отправка уведомлений о событиях
- **researcher** — поиск информации в интернете (в изолированном браузере)
- **coder** — помощь в программировании (с доступом к файловой системе в песочнице)

**Рассмотреть альтернативные архитектуры**

Если ваши требования к безопасности растут — например, для работы с финансовыми данными или в регулируемых отраслях — рассмотрите альтернативы:

- **IronClaw** (github.com/nearai/ironclaw) — переписанный на Rust вариант OpenClaw с песочницами WebAssembly (WASM — компактная виртуальная машина с нулевыми правами по умолчанию) и сертифицированной криптографией (FIPS 140-3).
