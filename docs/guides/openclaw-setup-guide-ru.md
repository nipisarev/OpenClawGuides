## Фаза 0: Что такое OpenClaw

### Что мы делаем и зачем

OpenClaw — это платформа для запуска личного AI-ассистента на собственном сервере. Вы пишете сообщение в Telegram или WhatsApp, оно попадает на ваш сервер, сервер пересылает запрос в AI-модель (Claude, GPT или другую), получает ответ и возвращает его вам в мессенджер.

Зачем это нужно, если можно просто открыть ChatGPT в браузере? Три причины:

**Приватность.** Ваши сообщения хранятся на вашем сервере. Наружу уходят только запросы к AI-модели — и больше ничего. Ни один посредник не видит вашу переписку.

**Доступность.** Ассистент работает 24/7 прямо в привычном мессенджере. Не нужно открывать отдельный сайт или приложение.

**Мульти-агентность.** На одном сервере можно запустить несколько AI-помощников с разными навыками. Один отвечает на вопросы, другой пишет код, третий читает и пересказывает документы — и каждый работает в изолированном контейнере (изолированная среда, как отдельная квартира в многоэтажке: соседи не могут зайти друг к другу).

### Схема

```
┌──────────┐         ┌──────────────────────────────────┐         ┌──────────────┐
│          │         │       Ваш VPS-сервер             │         │              │
│   Вы     │         │                                  │         │  AI-провайдер │
│          │  сооб-  │  ┌────────────────────────────┐  │ запрос  │  (Claude,    │
│ Telegram ├────────►│  │     OpenClaw Gateway       │  ├────────►│   GPT и др.) │
│   или    │  щение  │  │                            │  │         │              │
│ WhatsApp │◄────────┤  │  ┌────────┐  ┌────────┐   │  │◄────────┤              │
│          │  ответ  │  │  │Агент 1 │  │Агент 2 │   │  │  ответ  │              │
│          │         │  │  └────────┘  └────────┘   │  │         │              │
└──────────┘         │  └────────────────────────────┘  │         └──────────────┘
                     │                                  │
                     │  Сообщения остаются здесь.       │
                     │  Наружу уходят только запросы     │
                     │  к AI-модели.                    │
                     └──────────────────────────────────┘
```

**Gateway** (шлюз, центральный узел) — главный процесс OpenClaw. Он постоянно работает на сервере, принимает сообщения из мессенджеров, направляет их нужному агенту и возвращает ответы.

**Агент** — AI-помощник с заданной ролью. Каждый агент — это комбинация AI-модели, набора инструкций и списка разрешённых инструментов. Агенты работают в изолированных Docker-контейнерах (программные «коробки», которые не дают одному агенту влиять на другого или на сам сервер).

### Что понадобится

| Что | Зачем | Где взять |
|-----|-------|-----------|
| VPS-сервер (виртуальный сервер, который работает в дата-центре 24/7 — как компьютер, но в облаке) | Место, где будет жить OpenClaw | Hetzner, DigitalOcean, Aeza — см. Фазу 1 |
| Аккаунт Telegram | Мессенджер, через который вы будете общаться с ассистентом | telegram.org |
| API-ключ AI-провайдера (секретный код для доступа к AI-модели — как пароль от сервиса) | Чтобы ваш сервер мог отправлять запросы к Claude или GPT | console.anthropic.com или platform.openai.com |
| Примерно 30-40 минут | На всю установку от начала до работающего ассистента | — |

### Принцип безопасности

OpenClaw спроектирован по принципу «максимальная изоляция по умолчанию»:

- **Ваши сообщения** хранятся только на вашем сервере.
- **Наружу** уходят только запросы к AI-модели (Claude, GPT) — и ничего больше.
- **Gateway** слушает только на localhost (только «внутри» сервера — извне к нему подключиться нельзя).
- **Каждый агент** работает в отдельном Docker-контейнере с минимальными правами.
- **Удалённый доступ** — только через VPN (зашифрованный туннель, как секретный проход, которым пользуетесь только вы).

### Мульти-агентная архитектура

На одном сервере можно запустить несколько AI-помощников, каждый со своей специализацией:

```
┌──────────────────── Один VPS-сервер ────────────────────┐
│                                                          │
│  ┌─ OpenClaw Gateway ────────────────────────────────┐  │
│  │                                                    │  │
│  │  ┌──────────────┐  ┌──────────────┐               │  │
│  │  │  Ассистент   │  │   Кодер      │               │  │
│  │  │  Claude      │  │   GPT-4o     │               │  │
│  │  │  Общие       │  │   Пишет код  │               │  │
│  │  │  вопросы     │  │   в песочнице │               │  │
│  │  └──────────────┘  └──────────────┘               │  │
│  │                                                    │  │
│  │  ┌──────────────┐  ┌──────────────┐               │  │
│  │  │  Читатель    │  │  Исследова-  │               │  │
│  │  │  Claude      │  │  тель        │               │  │
│  │  │  Пересказ    │  │  GPT-4o      │               │  │
│  │  │  документов  │  │  Веб-поиск   │               │  │
│  │  └──────────────┘  └──────────────┘               │  │
│  │                                                    │  │
│  └────────────────────────────────────────────────────┘  │
│                                                          │
│  Telegram ──► Ассистент (по умолчанию)                  │
│               /coder ──► Кодер                           │
│  WhatsApp ──► Ассистент                                  │
└──────────────────────────────────────────────────────────┘
```

Каждый агент изолирован: у него свой API-ключ, свой набор инструментов и свой контейнер. Если один агент скомпрометирован — остальные не затронуты.

### Границы доверия

OpenClaw спроектирован как **персональный ассистент** — один Gateway = один пользователь. Любой авторизованный клиент получает полные права оператора над всем шлюзом. Разделения ролей внутри одного Gateway нет.

| Сценарий | Безопасность | Пояснение |
|----------|-------------|-----------|
| Один пользователь, несколько агентов | Безопасно | Штатный режим работы. Docker-изоляция + политики инструментов ограничивают ущерб |
| Два доверенных члена семьи | Умеренный риск | Нет изоляции сессий — каждый видит историю другого |
| Команда/организация на одном Gateway | Небезопасно | Нет разделения ролей, нет раздельного аудита, учётные данные видны всем |

Если вам нужен доступ для нескольких пользователей — разворачивайте отдельные Gateway для каждого.

## Фаза 1: Получаем сервер

### Что мы делаем и зачем

Нам нужен VPS (Virtual Private Server — виртуальный сервер в дата-центре, который работает круглосуточно, как удалённый компьютер, доступный через интернет). На нём будет жить OpenClaw.

Если у вас уже есть VPS с Ubuntu 22.04 или Debian 12 и вы можете подключиться к нему по SSH — переходите сразу к Фазе 2.

Если нет — следуйте шагам ниже.

### Схема

```
┌──────────────┐                              ┌──────────────────┐
│              │        SSH-соединение         │                  │
│  Ваш         │  (зашифрованный канал        │  VPS-сервер      │
│  компьютер   │   управления сервером)        │  Ubuntu 22.04    │
│              ├─────────────────────────────► │                  │
│  Терминал    │   ssh user@123.45.67.89      │  2 ГБ RAM        │
│              │                               │  20 ГБ SSD       │
└──────────────┘                              └──────────────────┘
```

### Минимальные требования к серверу

| Параметр | Минимум | Рекомендация |
|----------|---------|--------------|
| RAM (оперативная память) | 2 ГБ | 4 ГБ |
| CPU (процессор) | 1 ядро | 2 ядра |
| Диск | 20 ГБ SSD | 40 ГБ SSD |
| ОС (операционная система) | Ubuntu 22.04 или Debian 12 | Ubuntu 22.04 — проще для начинающих |

### Выбор провайдера

| Провайдер | Мин. цена | Расположение | Особенности |
|-----------|-----------|--------------|-------------|
| Hetzner | ~4 EUR/мес | Германия, Финляндия | Лучшее соотношение цена/качество, стабильная инфраструктура |
| DigitalOcean | ~6 USD/мес | США, Европа, Азия | Простой интерфейс, обширная документация на английском |
| Aeza | ~200 RUB/мес | Россия, Европа | Русскоязычная поддержка, оплата в рублях |

Для этого руководства подойдёт любой провайдер. Примеры ниже написаны в общем виде.

### Шаги

#### Шаг 1. Зарегистрируйтесь у выбранного провайдера

Перейдите на сайт провайдера и создайте аккаунт. Потребуется email и способ оплаты.

- Hetzner: https://www.hetzner.com/cloud
- DigitalOcean: https://www.digitalocean.com
- Aeza: https://aeza.net

#### Шаг 2. Сгенерируйте SSH-ключ

SSH-ключ — это цифровой пропуск для входа на сервер. Вместо пароля (который можно подобрать) используется пара файлов: публичный ключ (замок, который вы ставите на сервер) и приватный ключ (ключ, который остаётся только у вас). Подобрать такой «ключ» практически невозможно.

Откройте терминал (программу для ввода команд) на вашем компьютере и выполните команду:

**macOS и Linux** — откройте приложение «Терминал» (Terminal):

```bash
ssh-keygen -t ed25519 -C "openclaw-vps"
```

**Windows** — откройте Windows Terminal или PowerShell (нажмите Win, введите «PowerShell», откройте):

```powershell
ssh-keygen -t ed25519 -C "openclaw-vps"
```

На все вопросы нажимайте Enter (путь по умолчанию, пароль пустой для начала).

Ожидаемый результат:

```
Generating public/private ed25519 key pair.
Enter file in which to save the key (/Users/вашеимя/.ssh/id_ed25519):
Enter passphrase (empty for no passphrase):
Enter same passphrase again:
Your identification has been saved in /Users/вашеимя/.ssh/id_ed25519
Your public key has been saved in /Users/вашеимя/.ssh/id_ed25519.pub
The key fingerprint is:
SHA256:xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx openclaw-vps
```

Теперь выведите публичный ключ, чтобы скопировать его:

```bash
cat ~/.ssh/id_ed25519.pub
```

Ожидаемый результат — строка вида:

```
ssh-ed25519 AAAAC3Nza... openclaw-vps
```

Скопируйте эту строку целиком. Она понадобится на следующем шаге.

#### Шаг 3. Создайте сервер

В панели управления провайдера создайте новый сервер со следующими параметрами:

- **ОС:** Ubuntu 22.04
- **Тариф:** минимум 2 ГБ RAM, 1 vCPU, 20 ГБ SSD
- **Регион:** ближайший к вам (для минимальной задержки)
- **SSH-ключ:** вставьте публичный ключ из предыдущего шага (строку `ssh-ed25519 AAAAC3Nza... openclaw-vps`)
- **Имя сервера:** любое, например `openclaw`

После создания провайдер покажет IP-адрес (набор цифр вида `123.45.67.89`) вашего сервера. Запишите его.

#### Шаг 4. Подключитесь к серверу

SSH (Secure Shell) — зашифрованное соединение для удалённого управления сервером. Вы вводите команды на своём компьютере, а они выполняются на сервере.

Подождите 1-2 минуты после создания сервера (ему нужно время на запуск) и выполните:

```bash
ssh root@ваш_ip_адрес
```

Замените `ваш_ip_адрес` на IP, который вам показал провайдер. Например: `ssh root@123.45.67.89`.

При первом подключении система спросит:

```
The authenticity of host '123.45.67.89' can't be established.
ED25519 key fingerprint is SHA256:xxxxxxxxxxx.
Are you sure you want to continue connecting (yes/no/[fingerprint])?
```

Введите `yes` и нажмите Enter. Это нормальный вопрос при первом подключении — система запоминает «отпечаток» сервера.

Ожидаемый результат — приглашение командной строки сервера:

```
root@openclaw:~#
```

Вы внутри сервера.

#### Шаг 5. Обновите систему

Первое, что нужно сделать на свежем сервере — установить все обновления безопасности:

```bash
apt update && apt upgrade -y
```

`apt` — менеджер пакетов (программа для установки и обновления программ в Ubuntu/Debian). `update` загружает список обновлений, `upgrade` устанавливает их. Флаг `-y` означает «отвечать „да" на все вопросы автоматически».

Ожидаемый результат — список обновлённых пакетов, заканчивающийся чем-то вроде:

```
XX upgraded, XX newly installed, 0 to remove and 0 not upgraded.
```

#### Шаг 6. Создайте обычного пользователя

Сейчас вы работаете как `root` — пользователь с неограниченными правами. Это как ходить по дому с мастер-ключом от всех дверей: удобно, но опасно — любая ошибка или взлом может повредить всю систему.

Принцип наименьших привилегий: для повседневной работы используйте обычного пользователя, а `root`-права запрашивайте только когда нужно (через команду `sudo`).

Создайте нового пользователя:

```bash
adduser openclaw
```

Система попросит задать пароль и заполнить информацию о пользователе. Пароль задайте надёжный, остальные поля можно оставить пустыми (нажимайте Enter).

Ожидаемый результат:

```
Adding user `openclaw' ...
Adding new group `openclaw' (1001) ...
Adding new user `openclaw' (1001) with group `openclaw' ...
Creating home directory `/home/openclaw' ...
Copying files from `/etc/skel' ...
New password:
Retype new password:
passwd: password updated successfully
...
Is the information correct? [Y/n] Y
```

Дайте новому пользователю право выполнять команды от имени администратора через `sudo` (команда, которая позволяет обычному пользователю выполнить одну конкретную команду с правами root — вместо того чтобы быть root постоянно):

```bash
usermod -aG sudo openclaw
```

#### Шаг 7. Настройте SSH-доступ для нового пользователя

Скопируйте ваш SSH-ключ с root-пользователя на нового пользователя, чтобы вы могли входить на сервер напрямую под ним:

```bash
mkdir -p /home/openclaw/.ssh
cp /root/.ssh/authorized_keys /home/openclaw/.ssh/authorized_keys
chown -R openclaw:openclaw /home/openclaw/.ssh
chmod 700 /home/openclaw/.ssh
chmod 600 /home/openclaw/.ssh/authorized_keys
```

Что делает каждая команда:
- `mkdir -p` — создаёт папку `.ssh` в домашней директории нового пользователя
- `cp` — копирует файл с разрешёнными ключами от root к новому пользователю
- `chown -R` — передаёт владение файлами новому пользователю (иначе SSH откажет в доступе)
- `chmod 700` и `chmod 600` — устанавливают строгие права доступа к файлам (только владелец может читать и изменять)

#### Шаг 8. Отключитесь и подключитесь как новый пользователь

Выйдите из сервера:

```bash
exit
```

Подключитесь снова, но уже как пользователь `openclaw`:

```bash
ssh openclaw@ваш_ip_адрес
```

Ожидаемый результат:

```
openclaw@openclaw:~$
```

Обратите внимание: вместо `root@` теперь отображается `openclaw@` — вы вошли как обычный пользователь. Для команд, требующих права администратора, используйте `sudo` перед командой.

### ✓ Проверяем

Выполните следующую команду, чтобы убедиться, что всё работает:

```bash
whoami && sudo whoami
```

Ожидаемый результат:

```
openclaw
root
```

Первая строка — ваш текущий пользователь (`openclaw`). Вторая — подтверждение, что `sudo` работает (команда `whoami` выполнилась от имени `root`). При первом использовании `sudo` система попросит ввести пароль пользователя `openclaw`.

Если вы видите обе строки — сервер готов. Переходите к Фазе 2.

### Частые проблемы

**«Connection refused» при попытке подключения по SSH**

Сервер ещё не загрузился. Подождите 2-3 минуты после создания и попробуйте снова. Если не помогает — проверьте правильность IP-адреса в панели провайдера.

**«Permission denied (publickey)» при подключении**

SSH-ключ не был добавлен при создании сервера, или был добавлен неправильно. Решения:
- Убедитесь, что вы копировали именно публичный ключ (файл с расширением `.pub`).
- В панели провайдера удалите сервер, заново добавьте SSH-ключ и создайте сервер снова.
- Если провайдер позволяет — откройте веб-консоль сервера и добавьте ключ вручную.

**Не могу найти терминал на Windows**

Нажмите клавишу `Win` на клавиатуре, введите `PowerShell` и откройте найденную программу. Если у вас Windows 11 — можно также найти «Windows Terminal» (более удобный вариант). Все SSH-команды работают в обеих программах одинаково.

**«sudo: command not found» у нового пользователя**

На некоторых минимальных образах Debian `sudo` не предустановлен. Войдите как `root` и установите его:

```bash
ssh root@ваш_ip_адрес
apt install -y sudo
usermod -aG sudo openclaw
exit
```

Затем снова войдите как `openclaw`.

**Забыли пароль нового пользователя**

Войдите как `root` и сбросьте пароль:

```bash
ssh root@ваш_ip_адрес
passwd openclaw
exit
```

## Фаза 2: Защищаем сервер

### Что мы делаем и зачем

Сейчас ваш сервер похож на дом с открытыми дверьми — кто угодно может попробовать зайти. В этой фазе мы выстраиваем пять уровней защиты, чтобы даже при взломе одного уровня остальные продолжали работать. Это называется «эшелонированная оборона» (defense in depth — когда одна линия обороны не справляется, за ней стоит следующая).

### Схема

```
Интернет
    │
    ▼
┌──────────────────────────────────────────────────────────────────┐
│  Уровень 1: UFW-файрвол — блокирует всё, кроме SSH и Tailscale │
├──────────────────────────────────────────────────────────────────┤
│  Уровень 2: Fail2ban — банит IP после 5 неудачных попыток входа │
├──────────────────────────────────────────────────────────────────┤
│  Уровень 3: Tailscale VPN — приватная сеть, сервер невидим      │
├──────────────────────────────────────────────────────────────────┤
│  Уровень 4: SSH только по ключу — пароли полностью отключены    │
├──────────────────────────────────────────────────────────────────┤
│  Уровень 5: Автообновления — дыры в безопасности латаются сами  │
└──────────────────────────────────────────────────────────────────┘
    │
    ▼
Ваш сервер — защищён
```

### Шаги

#### Шаг 1. Настраиваем UFW-файрвол

Файрвол (firewall — «огненная стена») — это охранник на входе в ваш сервер. Он решает, какой сетевой трафик пропустить, а какой заблокировать. UFW (Uncomplicated Firewall) — упрощённая утилита для управления этим охранником.

**1.1. Запрещаем все входящие соединения по умолчанию:**

```bash
sudo ufw default deny incoming
```

Эта команда говорит файрволу: «блокируй вообще всё, что приходит снаружи, пока я не разрешу явно». Это самое безопасное начальное состояние.

Ожидаемый вывод:
```
Default incoming policy changed to 'deny'
```

**1.2. Разрешаем SSH-подключения:**

```bash
sudo ufw allow 22/tcp
```

Порт (port) — это пронумерованная «дверь» на сервере. Порт 22 — стандартная дверь для SSH. Если её не открыть, вы сами не сможете подключиться к серверу после включения файрвола.

Ожидаемый вывод:
```
Rules updated
Rules updated (v6)
```

**1.3. Разрешаем Tailscale:**

```bash
sudo ufw allow 41641/udp
```

Порт 41641 нужен для работы Tailscale VPN, который мы настроим в шаге 3. UDP — это тип сетевого протокола (в отличие от TCP он быстрее, но менее надёжен; для VPN это оптимальный выбор).

Ожидаемый вывод:
```
Rules updated
Rules updated (v6)
```

**1.4. Включаем файрвол:**

```bash
sudo ufw enable
```

Система спросит подтверждение — введите `y` и нажмите Enter.

Ожидаемый вывод:
```
Command may disrupt existing SSH connections. Proceed with operation (y|n)? y
Firewall is active and enabled on system startup
```

Файрвол теперь будет автоматически запускаться при каждой перезагрузке сервера.

---

#### Шаг 2. Устанавливаем Fail2ban

Fail2ban — это автоматический блокировщик. Он следит за журналами (логами) входа на сервер и, если кто-то пытается угадать ваш пароль несколько раз подряд, блокирует его IP-адрес (IP-адрес — уникальный номер каждого устройства в интернете, как почтовый индекс).

**2.1. Установка:**

```bash
sudo apt update && sudo apt install -y fail2ban
```

Ожидаемый вывод — длинный список строк, заканчивающийся на:
```
Setting up fail2ban ...
```

**2.2. Создаём файл настроек:**

Fail2ban хранит свои настройки в каталоге `/etc/fail2ban/`. Файл `jail.local` — это ваши пользовательские правила, которые не будут перезаписаны при обновлении программы.

```bash
sudo tee /etc/fail2ban/jail.local << 'EOF'
[sshd]
enabled = true
port = ssh
filter = sshd
maxretry = 5
bantime = 3600
findtime = 600
EOF
```

Что означают эти настройки:
- `enabled = true` — защита SSH включена
- `maxretry = 5` — после 5 неудачных попыток входа IP блокируется
- `bantime = 3600` — блокировка на 3600 секунд (1 час)
- `findtime = 600` — 5 попыток должны произойти в течение 600 секунд (10 минут), чтобы сработала блокировка

**2.3. Запускаем и включаем автозапуск:**

```bash
sudo systemctl enable --now fail2ban
```

Команда `systemctl` управляет службами (сервисами) операционной системы. Флаг `enable` включает автозапуск при перезагрузке, а `--now` запускает службу прямо сейчас.

Ожидаемый вывод:
```
Synchronizing state of fail2ban.service...
```

**2.4. Проверяем, что Fail2ban работает:**

```bash
sudo fail2ban-client status sshd
```

Ожидаемый вывод:
```
Status for the jail: sshd
|- Filter
|  |- Currently failed: 0
|  `- Total failed:     0
`- Actions
   |- Currently banned: 0
   `- Total banned:     0
```

#### Шаг 3. Настраиваем Tailscale VPN

Вместо того чтобы ваш сервер был виден всему интернету, VPN (Virtual Private Network — виртуальная частная сеть) создаёт приватный туннель. Представьте: сейчас ваш сервер стоит на оживлённой улице с табличкой «я здесь». С VPN вы прячете его в закрытый двор, а ключ от ворот есть только у ваших устройств.

Tailscale — самый простой VPN для нашей задачи. Он работает по технологии WireGuard и не требует сложной настройки.

**3.1. Создайте аккаунт Tailscale:**

Откройте в браузере https://login.tailscale.com/start и зарегистрируйтесь. Можно войти через Google, GitHub или Microsoft-аккаунт.

**3.2. Установите Tailscale на VPS:**

```bash
curl -fsSL https://tailscale.com/install.sh | sh
```

Эта команда скачивает и запускает установщик. `curl` — утилита для скачивания файлов из интернета. Флаги `-fsSL` означают: не показывать прогресс-бар, но показывать ошибки, и следовать за перенаправлениями.

Ожидаемый вывод заканчивается строкой:
```
Installation complete.
```

**3.3. Подключите VPS к вашей сети Tailscale:**

```bash
sudo tailscale up
```

Команда выведет ссылку для авторизации:
```
To authenticate, visit:

    https://login.tailscale.com/a/abc123...
```

Откройте эту ссылку в браузере на вашем компьютере и подтвердите подключение.

**3.4. Узнайте Tailscale-IP вашего сервера:**

```bash
tailscale ip -4
```

Ожидаемый вывод — IP-адрес вида:
```
100.x.y.z
```

Запишите этот адрес — по нему вы будете подключаться к серверу через VPN.

**3.5. Установите Tailscale на ваше устройство (компьютер или телефон):**

- **macOS/Windows/Linux:** скачайте с https://tailscale.com/download
- **iOS:** App Store, поиск «Tailscale»
- **Android:** Google Play, поиск «Tailscale»

После установки войдите в тот же аккаунт, что и на шаге 3.1.

**3.6. Проверьте подключение через VPN:**

С вашего устройства (не с самого VPS) подключитесь к серверу через Tailscale-IP:

```bash
ssh ваш_пользователь@100.x.y.z
```

Замените `100.x.y.z` на адрес из шага 3.4, а `ваш_пользователь` — на ваше имя пользователя на сервере.

Если подключение прошло успешно, вы попали на сервер через приватный туннель.

#### Шаг 4. Настраиваем автоматические обновления безопасности

Каждый день в программном обеспечении находят уязвимости. Автообновления (unattended-upgrades) — это как автоматическая замена замков, если в старых нашли дефект.

**4.1. Установка:**

```bash
sudo apt install -y unattended-upgrades
```

**4.2. Включаем автоматические обновления:**

```bash
sudo dpkg-reconfigure -plow unattended-upgrades
```

Система спросит: «Автоматически загружать и устанавливать стабильные обновления?» — выберите `Yes`.

**4.3. Проверяем, что автообновления работают:**

```bash
sudo unattended-upgrade --dry-run
```

Флаг `--dry-run` означает «пробный запуск» — команда покажет, что она *бы* обновила, но ничего не изменит. Если команда завершилась без ошибок, автообновления настроены верно.

#### Шаг 5. Отключаем вход по паролю через SSH

Сейчас вы входите на сервер по SSH-ключу (настроено в фазе 1). Но параллельно сервер по-прежнему принимает и парольный вход. Это лишняя дверь, которую нужно закрыть — пароли можно подобрать, ключи — практически нет.

**5.1. Откройте файл настроек SSH:**

```bash
sudo nano /etc/ssh/sshd_config
```

`nano` — простой текстовый редактор в терминале.

**5.2. Найдите и измените следующие строки:**

Используйте `Ctrl+W` для поиска. Найдите каждую из этих строк и убедитесь, что они выглядят именно так (без символа `#` в начале):

```
PasswordAuthentication no
PubkeyAuthentication yes
ChallengeResponseAuthentication no
```

Если строка начинается с `#` — удалите `#` (это знак комментария, он отключает настройку). Если значение стоит `yes` вместо `no` — замените.

Сохраните файл: `Ctrl+O`, затем `Enter`, затем `Ctrl+X` для выхода.

**5.3. Перезапустите SSH-службу:**

```bash
sudo systemctl restart sshd
```

Важно: не закрывайте текущую SSH-сессию! Откройте новый терминал и проверьте, что вы можете подключиться по ключу. Если что-то пошло не так, в первом терминале вы сможете откатить изменения.

**5.4. Проверьте, что парольный вход отключён:**

В новом терминале попробуйте подключиться с принудительным использованием пароля:

```bash
ssh -o PubkeyAuthentication=no ваш_пользователь@IP_вашего_сервера
```

Ожидаемый вывод:
```
Permission denied (publickey).
```

Это означает, что сервер отказал в доступе по паролю — именно то, что нужно.

#### Шаг 6. Защита конфигурационных файлов от изменения

После того как SSH-настройки зафиксированы, мы ставим на них «замок на уровне ядра ОС» — атрибут `chattr +i` (immutable — неизменяемый). Этот атрибут запрещает изменение файла даже пользователю root, как если бы вы повесили на дверь замок, ключ от которого нужно явно попросить каждый раз.

**6.1. Сделайте конфигурационные файлы неизменяемыми:**

```bash
sudo chattr +i /etc/ssh/sshd_config
sudo chattr +i /root/.ssh/authorized_keys
```

Теперь эти файлы нельзя изменить, удалить или переименовать — ни вручную, ни через скрипт, ни через вредоносную программу.

**6.2. Как редактировать защищённые файлы**

Если в будущем вам понадобится изменить один из этих файлов, сначала снимите защиту, внесите изменения, затем верните защиту обратно:

```bash
sudo chattr -i /etc/ssh/sshd_config
# ... внесите изменения ...
sudo chattr +i /etc/ssh/sshd_config
```

**Важно:** Не используйте `chattr +i` на конфигурационных файлах OpenClaw (`openclaw.json`, `paired.json` и др.) — они должны быть доступны для записи во время работы программы. Для их защиты используйте `chmod 600` (доступ только для владельца) и контроль целостности через хеш-суммы (baseline).

### Частые проблемы

**Проблема: Потерял доступ к серверу после включения UFW**

Если вы случайно заблокировали себе SSH-доступ:
1. Войдите на сервер через **веб-консоль** вашего VPS-провайдера (Hetzner, DigitalOcean и т.д. — ищите кнопку «Console» или «VNC» в панели управления)
2. Выполните:
```bash
sudo ufw disable
sudo ufw allow 22/tcp
sudo ufw enable
```

Веб-консоль работает напрямую, минуя сеть, поэтому файрвол на неё не влияет.

**Проблема: Tailscale не подключается**

1. Проверьте, что на обоих устройствах (VPS и ваш компьютер/телефон) вы вошли в один и тот же аккаунт Tailscale
2. На VPS проверьте статус:
```bash
sudo systemctl status tailscaled
```
Должно быть `active (running)`. Если нет:
```bash
sudo systemctl restart tailscaled
sudo tailscale up
```
3. Убедитесь, что порт 41641/udp открыт в UFW (шаг 1.3)

**Проблема: Fail2ban заблокировал мой IP**

Если вы сами несколько раз ошиблись с паролем и были заблокированы:
```bash
sudo fail2ban-client set sshd unbanip ваш_IP_адрес
```

Чтобы узнать, какие IP заблокированы:
```bash
sudo fail2ban-client status sshd
```

**Проблема: После отключения паролей не могу войти**

Если парольный вход отключён, а ключ не работает:
1. Войдите через веб-консоль VPS-провайдера
2. Верните парольный вход:
```bash
sudo nano /etc/ssh/sshd_config
```
Измените `PasswordAuthentication no` на `PasswordAuthentication yes`, сохраните и выполните:
```bash
sudo systemctl restart sshd
```
3. Заново настройте SSH-ключ (по инструкции из фазы 1), затем повторите шаг 5

## Фаза 3: Устанавливаем OpenClaw

### Что мы делаем и зачем

Сервер защищён — теперь устанавливаем само приложение. OpenClaw — это платформа-шлюз (Gateway), которая связывает мессенджеры (Telegram, WhatsApp) с AI-моделями (Claude, GPT). Ваши сообщения обрабатываются на вашем сервере, наружу уходят только запросы к AI.

Есть два пути установки:
1. **Через Ansible** (рекомендуется) — автоматическая установка одной командой
2. **Вручную** — для тех, кто хочет понимать каждый шаг

Оба пути приводят к одинаковому результату.

### Схема

```
Ваш компьютер                          VPS
┌──────────────────┐                   ┌──────────────────────────────┐
│                  │   SSH/Tailscale   │                              │
│  Ansible-плейбук ├──────────────────►│  Node.js 22.x               │
│  (автоустановщик)│   устанавливает   │  pnpm (менеджер пакетов)    │
│                  │                   │  Docker CE (контейнеры)      │
└──────────────────┘                   │  OpenClaw Gateway            │
                                       │  systemd-служба              │
                                       └──────────────────────────────┘

Либо всё то же самое — вручную, команда за командой.
```

### Шаги

---

### Путь A: Автоматическая установка через Ansible (рекомендуется)

Ansible (читается «ансибл») — это автоматический установщик. Представьте робота-повара, которому вы даёте рецепт (плейбук — playbook), и он готовит блюдо шаг за шагом, ничего не пропуская. Ansible запускается на **вашем компьютере** и сам подключается к серверу по SSH, чтобы всё настроить.

#### A1. Установите Ansible на ваш компьютер (не на VPS!)

**macOS:**

```bash
brew install ansible
```

`brew` — менеджер пакетов для macOS (если не установлен, см. https://brew.sh).

**Ubuntu/Debian (если ваш локальный компьютер на Linux):**

```bash
sudo apt update && sudo apt install -y ansible
```

Проверьте установку:

```bash
ansible --version
```

Ожидаемый вывод — строка, начинающаяся с:
```
ansible [core 2.x.x]
```

#### A2. Скачайте Ansible-плейбук для OpenClaw

```bash
git clone https://github.com/openclaw/openclaw-ansible.git
cd openclaw-ansible
```

#### A3. Настройте файл инвентаря

Инвентарь (inventory) — это файл, в котором Ansible находит информацию о вашем сервере: его адрес, способ подключения и переменные для настройки.

```bash
cp inventory.example inventory
nano inventory
```

Заполните следующие поля:

```ini
[openclaw]
ваш_сервер ansible_host=100.x.y.z ansible_user=ваш_пользователь ansible_ssh_private_key_file=~/.ssh/id_ed25519

[openclaw:vars]
tailscale_auth_key=tskey-auth-xxxxx
```

Где:
- `100.x.y.z` — Tailscale-IP вашего сервера (из фазы 2, шаг 3.4)
- `ваш_пользователь` — имя пользователя на сервере
- `~/.ssh/id_ed25519` — путь к вашему SSH-ключу (из фазы 1)
- `tskey-auth-xxxxx` — авторизационный ключ Tailscale (получите на https://login.tailscale.com/admin/settings/keys, нажмите «Generate auth key»)

Сохраните: `Ctrl+O`, `Enter`, `Ctrl+X`.

#### A4. Запустите плейбук

```bash
ansible-playbook -i inventory site.yml
```

Эта команда подключится к вашему серверу и автоматически установит:
- **Node.js 22.x** — среда выполнения JavaScript, на которой написан OpenClaw
- **pnpm** — менеджер пакетов (программа для установки других программ из реестра npm, как apt для JavaScript-мира)
- **Docker CE** — платформа контейнеризации (о контейнерах подробнее ниже)
- **systemd-служба** — менеджер служб операционной системы, который держит OpenClaw запущенным, даже если сервер перезагрузится

Процесс займёт 5-10 минут. Вы увидите задачи, выполняемые одна за другой:

```
PLAY [openclaw] ****
TASK [Gathering Facts] **** ok
TASK [Install Node.js 22.x] **** changed
TASK [Install pnpm] **** changed
TASK [Install Docker CE] **** changed
...
PLAY RECAP ****
ваш_сервер : ok=15   changed=12   unreachable=0   failed=0
```

Если в строке `PLAY RECAP` значение `failed=0` — всё установлено успешно.

**После завершения плейбука подключитесь к серверу** и запустите онбординг:

```bash
ssh ваш_пользователь@100.x.y.z
openclaw onboard --install-daemon
```

### Общие шаги (для обоих путей)

#### Шаг 6. Применяем защищённую конфигурацию

Конфигурация (config) — файл с настройками поведения программы. OpenClaw использует формат JSON5 (расширенная версия JSON, которая поддерживает комментарии).

Откройте файл конфигурации:

```bash
nano ~/.openclaw/openclaw.json5
```

Замените содержимое на следующее:

```json5
{
  "$schema": "https://openclaw.ai/schemas/2024-11/config.json",
  "version": 1,

  "gateway": {
    "bind": "loopback",
    "port": 18789,
    "auth": {
      "mode": "token",
      "token": "СЮДА_ВСТАВЬТЕ_ТОКЕН_ИЗ_ШАГА_7"
    },
    "trustedProxies": [],
    "controlUi": {
      "dangerouslyDisableDeviceAuth": false
    }
  },

  "agents": {
    "defaults": {
      "tools": {
        "profile": "minimal"
      },
      "sandbox": {
        "mode": "all",
        "workspaceAccess": "none"
      }
    }
  },

  "logging": {
    "redactSensitive": "tools"
  },

  "browser": {
    "evaluateEnabled": false
  },

  "discovery": {
    "mdns": {
      "mode": "off"
    }
  },

  "plugins": {
    "enabled": false
  },

  "commands": {
    "config": false
  }
}
```

Что делает каждая настройка:

| Настройка | Значение | Зачем |
|-----------|----------|-------|
| `bind: "loopback"` | Gateway слушает только на `127.0.0.1` | Доступ только через Tailscale, не из открытого интернета |
| `auth.mode: "token"` | Вход по токену | Без токена никто не получит доступ к Gateway |
| `sandbox.mode: "all"` | Все инструменты агентов работают внутри Docker-контейнеров | Даже если агент сделает что-то опасное, это не выйдет за пределы контейнера |
| `sandbox.workspaceAccess: "none"` | Агенты не имеют доступа к файловой системе | Нет риска чтения или изменения файлов на сервере |
| `tools.profile: "minimal"` | Агенты получают минимальный набор инструментов | Меньше инструментов — меньше возможностей для злоупотребления |
| `redactSensitive: "tools"` | Чувствительные данные удаляются из логов | Токены и ключи не попадут в журнал |
| `browser.evaluateEnabled: false` | Запуск JavaScript в браузере отключён | Закрывает вектор инъекции кода |
| `discovery.mdns.mode: "off"` | Сервер не анонсирует себя в сети | VPS не нужно обнаружение в локальной сети |
| `plugins.enabled: false` | Плагины отключены | Никакой сторонний код не запустится, пока вы явно не разрешите |
| `commands.config: false` | Изменение конфигурации через чат запрещено | Настройки меняются только через терминал на сервере |

Не закрывайте файл — сначала сгенерируйте токен (шаг 7), вставьте его и только потом сохраните.

#### Шаг 6.5. Ограничиваем ресурсы Gateway

В Node.js (среда выполнения, на которой работает OpenClaw) существует известная проблема с утечкой памяти (Issue #13758 — Gateway может потреблять до 1.9 ГБ памяти после 13 часов непрерывной работы). Утечка памяти (memory leak) — это когда программа запрашивает у системы память, но не возвращает её обратно, как если бы вы брали тарелки из шкафа, но никогда не ставили их обратно — рано или поздно шкаф опустеет.

Чтобы Gateway не «съел» всю память на сервере, ограничим его ресурсы через systemd:

```bash
# Add Node.js memory limit to systemd service
sudo mkdir -p /etc/systemd/system/openclaw.service.d
sudo tee /etc/systemd/system/openclaw.service.d/resources.conf << 'EOF'
[Service]
Environment=NODE_OPTIONS=--max-old-space-size=1536
MemoryHigh=1536M
MemoryMax=2G
Restart=always
RestartSec=5
EOF
sudo systemctl daemon-reload
sudo systemctl restart openclaw
```

Что делает каждая настройка:

| Настройка | Значение | Зачем |
|-----------|----------|-------|
| `--max-old-space-size=1536` | Ограничивает кучу Node.js до 1536 МБ | Предотвращает бесконтрольный рост памяти |
| `MemoryHigh=1536M` | Мягкий лимит — система начинает активно освобождать память | Замедляет процесс вместо его убийства |
| `MemoryMax=2G` | Жёсткий лимит — процесс будет перезапущен при превышении | Защищает сервер от зависания |
| `Restart=always` | Автоматический перезапуск при любом сбое | Gateway восстановится без вашего вмешательства |

#### Шаг 6.6. Защищаем директорию с секретами

Директория `~/.openclaw/` хранит OAuth-токены (токены авторизации для подключённых сервисов), сессионные ключи и другие секреты в открытом виде. Если другой пользователь на сервере получит к ним доступ, он сможет управлять вашим ассистентом.

```bash
# Защищаем директорию с секретами
chmod 700 ~/.openclaw/
```

Права доступа `700` означают: только владелец может читать, записывать и входить в эту директорию. Все остальные пользователи системы не увидят даже список файлов внутри — как запертая комната, ключ от которой есть только у вас.

#### Шаг 6.7. Настраиваем изоляцию сессий

По умолчанию OpenClaw объединяет контекст всех пользователей, которые пишут боту, в одну сессию (dmScope — область видимости личных сообщений). Это значит, что секреты или файлы, загруженные одним пользователем, могут быть прочитаны другим. Даже если вы единственный пользователь, рекомендуется установить изоляцию:

```bash
openclaw config set session.dmScope per-channel-peer
```

Значение `per-channel-peer` создаёт отдельную сессию для каждой комбинации «канал + собеседник». Ваши сообщения в Telegram не смешиваются с сообщениями в WhatsApp, и если кто-то другой напишет вашему боту — он не увидит вашу переписку.

#### Шаг 6.8. Ограничиваем инструменты в групповых чатах

Если ваш бот добавлен в групповой чат, по умолчанию все участники группы могут использовать его инструменты — включая выполнение команд и доступ к файлам. Это опасно.

Добавьте в конфигурацию ограничения для групп. Откройте `~/.openclaw/openclaw.json5` и в блок `channels` добавьте:

```json5
"channels": {
  "telegram": {
    "linkPreview": false,
    "groups": {
      "*": {
        "requireMention": true,
        "tools": {
          "allow": ["read", "message"],
          "deny": ["exec", "write", "edit", "browser", "gateway", "nodes"]
        }
      }
    }
  }
}
```

Что это делает:
- `"*"` — правило применяется ко **всем** группам
- `requireMention: true` — бот реагирует только когда его упоминают (@имя_бота), а не на каждое сообщение
- `deny` — запрещает выполнение команд (`exec`), запись файлов (`write`, `edit`), браузер (`browser`) и управление шлюзом (`gateway`, `nodes`)
- Остаётся только чтение (`read`) и отправка сообщений (`message`)

#### Шаг 7. Генерируем токен авторизации

Токен (token) — это длинная случайная строка, которая служит паролем для доступа к Gateway. В отличие от обычного пароля, его невозможно угадать.

```bash
openssl rand -hex 32
```

Ожидаемый вывод — строка из 64 символов (букв и цифр):
```
a1b2c3d4e5f6...  (64 символа)
```

Скопируйте эту строку и вставьте в конфигурацию вместо `СЮДА_ВСТАВЬТЕ_ТОКЕН_ИЗ_ШАГА_7`.

Сохраните файл конфигурации: `Ctrl+O`, `Enter`, `Ctrl+X`.

Важно: сохраните токен в надёжном месте (менеджер паролей). Если потеряете — придётся генерировать новый.

#### Шаг 7.1. Защита токена авторизации

OpenClaw может передавать токен через URL-параметры (`?token=...`), что приводит к утечке через историю браузера, заголовки Referer и логи прокси-серверов.

**Правила защиты токена:**

1. **Длина токена** — используйте только `openssl rand -hex 32` (256 бит). OpenClaw не проверяет длину токена — слабый токен вроде `1234` будет принят, но это критическая уязвимость
2. **Не отключайте аутентификацию устройств** — настройка `dangerouslyDisableDeviceAuth: false` добавляет второй уровень защиты через одноразовые коды сопряжения (pairing code — короткоживущий код, который Gateway показывает в терминале при первом подключении нового устройства)
3. **Очищайте историю браузера** после первого доступа к Control UI
4. **Меняйте токен регулярно** — раз в месяц или при подозрении на утечку:

```bash
openclaw config set gateway.auth.token $(openssl rand -hex 32)
```

5. **Никогда не сохраняйте URL с токеном** в закладках и не передавайте его через мессенджеры

#### Шаг 8. Ставим конфигурацию под контроль версий

Контроль версий (version control) — система, которая запоминает каждое изменение в файлах. Если кто-то (или что-то) изменит вашу конфигурацию, вы сможете увидеть, что именно поменялось, и откатить изменения.

```bash
cd ~/.openclaw && git init && git add . && git commit -m "baseline"
```

Эта команда:
1. Переходит в каталог настроек OpenClaw
2. Инициализирует репозиторий Git (систему контроля версий)
3. Добавляет все файлы
4. Создаёт первый «снимок» — точку отсчёта

Ожидаемый вывод заканчивается на:
```
[main (root-commit) xxxxxxx] baseline
 X files changed, Y insertions(+)
```

В дальнейшем вы сможете проверять, не изменилась ли конфигурация:

```bash
cd ~/.openclaw && git diff
```

Если вывод пустой — ничего не изменилось. Если есть изменения — вы увидите, что именно поменялось.

### Частые проблемы

**Проблема: Docker не запускается**

Проверьте статус службы:
```bash
sudo systemctl status docker
```

Если статус не `active (running)`:
```bash
sudo systemctl start docker
sudo systemctl enable docker
```

Если Docker выдаёт ошибку `permission denied`:
```bash
sudo usermod -aG docker $USER
```

После этого **обязательно** перезайдите на сервер (выйдите и подключитесь снова).

**Проблема: Неправильная версия Node.js**

```bash
node -v
```

Если версия не начинается с `v22`:
```bash
curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash -
sudo apt install -y nodejs
```

**Проблема: `openclaw` команда не найдена после установки**

Если терминал пишет `command not found: openclaw`:
```bash
export PATH="$PATH:$(pnpm bin -g)"
```

Чтобы изменение сохранилось после перезахода:
```bash
echo 'export PATH="$PATH:$(pnpm bin -g)"' >> ~/.bashrc
source ~/.bashrc
```

**Проблема: Permission denied при запуске OpenClaw**

Убедитесь, что вы работаете под обычным пользователем, а не под root. OpenClaw специально не запускается от root из соображений безопасности:
```bash
whoami
```

Вывод должен быть именем вашего пользователя, **не** `root`. Если вы под root — выйдите и зайдите под обычным пользователем.

**Проблема: `openclaw security audit` показывает предупреждения**

Попробуйте автоматическое исправление безопасных проблем:
```bash
openclaw security audit --fix
```

Команда исправит то, что можно исправить автоматически, и покажет оставшиеся проблемы, которые нужно решить вручную.

## Фаза 4: Первый агент — быстрый старт

### Что мы делаем и зачем

В предыдущих фазах мы подготовили сервер: установили OpenClaw, настроили защиту и VPN-доступ. Но пока система молчит — к ней не подключен ни один мессенджер и ни одна AI-модель.

В этой фазе мы получим первый рабочий результат: отправим сообщение боту в Telegram и получим ответ от AI. Для этого нужно сделать три вещи: получить ключ доступа к AI-сервису, создать Telegram-бота и связать всё это внутри OpenClaw.

### Схема

```
                        Интернет
                           │
Вы (Telegram) ───► Telegram Bot API ───► OpenClaw Gateway ───► Claude/GPT API
                                          (ваш сервер)          (AI-сервис)
      ◄── ответ ◄──────────────────── ◄── ответ ◄─────────── ◄── ответ ◄──
```

### Шаги

**Шаг 1. Получите API-ключ от AI-сервиса**

API-ключ (Application Programming Interface key) — это пароль для доступа к AI-сервису, как пароль от Wi-Fi: без него ваш сервер не сможет отправлять запросы к Claude или GPT.

Выберите один из вариантов:

*Вариант A — Claude (Anthropic):*

1. Откройте https://console.anthropic.com
2. Зарегистрируйтесь и подтвердите email
3. Перейдите в раздел "API Keys"
4. Привяжите способ оплаты (карта) — AI-сервисы тарифицируются по количеству обработанного текста, стоимость обычного диалога — несколько центов
5. Нажмите "Create Key", дайте ключу понятное имя, например `openclaw-vps`
6. Скопируйте ключ — он показывается только один раз

*Вариант B — GPT (OpenAI):*

1. Откройте https://platform.openai.com
2. Зарегистрируйтесь и подтвердите email
3. Перейдите в раздел "API keys"
4. Привяжите способ оплаты
5. Нажмите "Create new secret key", дайте имя `openclaw-vps`
6. Скопируйте ключ

Рекомендация по безопасности: установите лимит расходов (spending limit) в настройках аккаунта провайдера. Начните с $5-10 в месяц — для личного использования этого достаточно.

ВАЖНО: никогда не отправляйте API-ключ в чаты, не храните его в заметках или документах с общим доступом. Относитесь к нему как к паролю от банковского приложения.

**Шаг 2. Создайте Telegram-бота**

Telegram-бот — это специальный аккаунт в Telegram, который управляется программой (в нашем случае — OpenClaw), а не человеком. Вы будете писать ему сообщения, а отвечать будет AI.

Бот создаётся через специального бота `@BotFather` — это официальный инструмент Telegram для управления ботами.

1. Откройте Telegram и найдите `@BotFather` (с синей галочкой верификации)
2. Отправьте ему команду:

```
/newbot
```

3. BotFather спросит имя бота — это отображаемое имя, например `Мой AI Помощник`
4. Затем спросит username — это техническое имя, должно заканчиваться на `bot`, например `my_ai_helper_bot`
5. BotFather ответит сообщением с токеном (token) бота — строка вида `7123456789:AAF...`

Токен бота — это адрес и пароль вашего бота в одной строке. Кто владеет токеном — тот управляет ботом. Сохраните его так же надёжно, как API-ключ.

**Шаг 3. Подключите Telegram-канал к OpenClaw**

Ожидаемый результат: OpenClaw запустит интерактивный мастер настройки. Он попросит ввести токен бота.

Далее настройте список допущенных пользователей — allowlist (список тех, кому разрешено общаться с ботом). Это важно: без allowlist любой пользователь Telegram сможет писать вашему боту и тратить ваш API-бюджет.


Ожидаемый результат:

**Шаг 3.5. Отключите предпросмотр ссылок**

**Шаг 4. Настройте AI-модель и безопасность агента**

Установите минимальный набор инструментов — агент сможет только вести диалог, без доступа к файлам, браузеру или командной строке:

```bash
openclaw config set agents.defaults.tools.profile minimal
```

### Частые проблемы

**«Бот не отвечает, в логах тишина»**

Токен бота введён неверно или Gateway не запущен. Проверьте токен: `openclaw channels status`. Перезапустите: `openclaw gateway restart`.

**«Unauthorized» или «401» в логах**

API-ключ неверный, просрочен или не активирован. Проверьте ключ в личном кабинете провайдера (console.anthropic.com или platform.openai.com). Убедитесь, что привязан способ оплаты.

**«Ответ приходит через 30+ секунд»**

Нормально для первого сообщения — AI-сервис «прогревается». Также зависит от загрузки серверов провайдера. Подождите. Последующие сообщения будут быстрее.

**«Rate limited» или «429» в логах**

Вы отправляете слишком много сообщений за короткое время. У каждого провайдера есть лимит запросов в минуту. Подождите минуту и попробуйте снова. Для новых аккаунтов лимиты ниже.

**«Insufficient funds» или «402»**

Баланс API-аккаунта исчерпан. Пополните баланс в личном кабинете провайдера.

**«Бот отвечает, но не вам»**

Allowlist настроен неверно, и кто-то другой нашёл вашего бота. Пересмотрите allowlist: `openclaw channels add telegram` (повторная настройка).

## Фаза 5: Команда агентов

### Что мы делаем и зачем

В Фазе 4 мы запустили одного универсального агента. Он может вести диалог, но не умеет писать код, работать с файлами или искать информацию в интернете. Давать одному агенту все возможности сразу — плохая идея с точки зрения безопасности: если AI ошибётся или будет обманут специально подготовленным сообщением (prompt injection — атака через текст, заставляющая AI выполнить нежелательные действия), он получит доступ ко всему сразу.

Представьте, что у вас есть команда помощников. Каждый — специалист в своей области. Один отвечает на вопросы, другой пишет код, третий ищет информацию в интернете. Все работают на одном сервере, но изолированы друг от друга — как сотрудники в разных кабинетах с разными ключами доступа.

В этой фазе мы настроим трёх агентов:

| Агент | Назначение | Что умеет | Чего НЕ умеет |
|-------|-----------|-----------|---------------|
| **Ассистент** | Повседневные вопросы, диалог | Только разговор | Файлы, код, интернет |
| **Кодер** | Код, файлы, технические задачи | Читать/писать файлы, выполнять команды | Интернет, браузер |
| **Исследователь** | Поиск информации в интернете | Браузер (в изоляции) | Файлы, выполнение команд |

### Схема

```
Telegram ──► OpenClaw Gateway (шлюз)
               │
               │  сообщение ──► маршрутизация по команде
               │
               ├── (по умолчанию) ──► Ассистент (Claude, только диалог)
               │                        └── Sandbox A (изолирован, нет инструментов)
               │
               ├── /coder ──────────► Кодер (GPT-4o, файлы + код)
               │                        └── Sandbox B (изолирован, доступ к /workspaces/coder)
               │
               └── /research ───────► Исследователь (Claude, браузер)
                                        └── Sandbox C (изолирован, нет доступа к файлам)

Каждый агент = отдельный Docker-контейнер + свой API-ключ + свои права
```

### Шаги

**Шаг 1. Создайте рабочую директорию для Кодера**

Рабочая директория (workspace) — это папка на сервере, куда агент-кодер сможет сохранять файлы. Агент увидит только эту папку и ничего за её пределами.

```bash
sudo mkdir -p /opt/openclaw/workspaces/coder
sudo chown openclaw:openclaw /opt/openclaw/workspaces/coder
```

Ожидаемый результат: команды выполнятся без ошибок. Проверьте:

```bash
ls -la /opt/openclaw/workspaces/
```

Ожидаемый результат:

```
drwxr-xr-x 2 openclaw openclaw 4096 ... coder
```

---

**Шаг 2. Получите дополнительные API-ключи (если нужно)**

Если вы хотите использовать разных AI-провайдеров для разных агентов (например, Claude для ассистента и GPT-4o для кодера), вам понадобятся API-ключи от обоих сервисов. Создайте их по инструкции из Фазы 4, Шаг 1.

Если вы хотите использовать одного провайдера для всех агентов — достаточно одного ключа. Но рекомендуется создать отдельные ключи для каждого агента: это позволит отслеживать расходы по агентам раздельно и отключить один ключ, не затрагивая остальных.

---

**Шаг 3. Сохраните API-ключи в переменных окружения**

Переменные окружения (environment variables) — это именованные значения, доступные программам на сервере. Они безопаснее, чем запись ключей напрямую в файл конфигурации, потому что не попадут в историю команд или резервные копии конфигурации.

Откройте файл окружения для OpenClaw:

```bash
sudo nano /etc/openclaw/env
```

Добавьте строки с вашими ключами (замените значения на реальные):

```
AGENT_ASSISTANT_ANTHROPIC_KEY=sk-ant-ваш-ключ-для-ассистента
AGENT_CODER_OPENAI_KEY=sk-ваш-ключ-для-кодера
AGENT_RESEARCHER_ANTHROPIC_KEY=sk-ant-ваш-ключ-для-исследователя
```

Сохраните файл (Ctrl+O, Enter, Ctrl+X) и ограничьте доступ к нему:

```bash
sudo chmod 600 /etc/openclaw/env
sudo chown openclaw:openclaw /etc/openclaw/env
```

`chmod 600` означает: только владелец файла может его читать и редактировать. Никто другой на сервере не получит доступ к ключам.

Чтобы OpenClaw загружал эти переменные при запуске, укажите путь к файлу в systemd-конфигурации:

```bash
sudo mkdir -p /etc/systemd/system/openclaw.service.d
sudo tee /etc/systemd/system/openclaw.service.d/env.conf << 'EOF'
[Service]
EnvironmentFile=/etc/openclaw/env
EOF
sudo systemctl daemon-reload
```

Команда `daemon-reload` заставляет systemd перечитать настройки служб. После этого при каждом запуске OpenClaw переменные из файла `/etc/openclaw/env` будут автоматически загружены.

---

#### Шаг 2.5. Правила безопасности для учётных данных

Подключение OpenClaw к внешним сервисам (GitHub, Slack, Google Workspace, календарь) создаёт так называемый «теневой ИИ» (Shadow AI — неконтролируемый агент с унаследованными правами пользователя). Если агент скомпрометирован через инъекцию промпта, злоумышленник наследует **все** права подключённых сервисов.

**Правила:**

1. **Только выделенные сервисные аккаунты** — никогда не используйте личные или рабочие учётные записи для предоставления доступа агентам
2. **Минимальные права** — GitHub: токены с доступом только к конкретным репозиториям, только на чтение. Slack: только `channels:read` + `chat:write`, никогда admin-права. Google: `gmail.readonly`, не `mail.google.com`
3. **Короткоживущие токены** — по возможности используйте токены с временем жизни 5 минут (сокращает риск кражи на 92% по данным Okta 2025)
4. **Любые учётные данные, попавшие в контекст AI-модели, считаются скомпрометированными** — меняйте их сразу

---

**Шаг 4. Создайте конфигурацию с тремя агентами**

Создайте файл конфигурации Gateway — это главный файл, описывающий всех агентов, их права и маршрутизацию сообщений.

> **Важно:** В Фазе 3 мы создали файл `openclaw.json5` с общими настройками безопасности (привязка к loopback, токен авторизации, режим песочницы). Файл `gateway.yaml` дополняет его — здесь мы описываем конкретных агентов, их модели и маршрутизацию. Оба файла живут в `~/.openclaw/` и работают вместе: `openclaw.json5` задаёт глобальные правила безопасности, а `gateway.yaml` — индивидуальные настройки каждого агента.

```bash
nano ~/.openclaw/gateway.yaml
```

Вставьте следующее содержимое целиком:

```yaml
# Главный файл конфигурации OpenClaw Gateway
# Описывает всех агентов, их права доступа и маршрутизацию сообщений

gateway:
  bind: loopback     # Gateway слушает только на localhost (127.0.0.1)
  port: 18789
  auth:
    mode: token      # Доступ к Gateway только по токену

# Настройки агентов
agents:

  # Общие настройки по умолчанию для всех агентов
  defaults:
    sandbox:
      mode: all          # Все инструменты работают внутри Docker
      scope: dedicated   # Каждый агент — отдельный контейнер
      docker: true
    tools:
      profile: minimal   # По умолчанию — минимум инструментов
    workspaceAccess: none # По умолчанию — без доступа к файлам

  # Описание каждого агента
  definitions:

    # Ассистент — повседневные вопросы, диалог
    # Нет инструментов, нет доступа к файлам
    assistant:
      model:
        provider: anthropic
        name: claude-sonnet-4-20250514
      tools:
        profile: minimal
      sandbox:
        mode: all
        scope: dedicated
        workspaceAccess: none
      credentials:
        anthropicApiKey: ${AGENT_ASSISTANT_ANTHROPIC_KEY}

    # Кодер — генерация кода, работа с файлами
    # Может читать и записывать файлы, выполнять команды
    # НО только в своей рабочей директории, без доступа к интернету
    coder:
      model:
        provider: openai
        name: gpt-4o
      tools:
        profile: coding
        allow: [file_read, file_write, shell_exec]
        deny: [browser, network_fetch]
      sandbox:
        mode: all
        scope: dedicated
        workspaceAccess: rw
        workspaceDir: /opt/openclaw/workspaces/coder
      credentials:
        openaiApiKey: ${AGENT_CODER_OPENAI_KEY}

    # Исследователь — поиск информации в интернете
    # Может открывать веб-страницы в изолированном браузере
    # НО не может записывать файлы или выполнять команды
    researcher:
      model:
        provider: anthropic
        name: claude-sonnet-4-20250514
      tools:
        profile: minimal
        allow: [browser]
        deny: [file_write, shell_exec]
      sandbox:
        mode: all
        scope: dedicated
        workspaceAccess: none
      credentials:
        anthropicApiKey: ${AGENT_RESEARCHER_ANTHROPIC_KEY}

# Каналы связи
channels:
  telegram:
    defaultAgent: assistant    # Обычные сообщения идут Ассистенту
    agentRouting:              # Команды для переключения на других агентов
      /coder: coder
      /research: researcher
```

Сохраните файл (Ctrl+O, Enter, Ctrl+X).

**Что здесь происходит:**

- `defaults` задаёт правила безопасности, которые применяются ко всем агентам, если для конкретного агента не указано иное
- Каждый агент в `definitions` описывает: какую AI-модель использовать, какие инструменты разрешены/запрещены, какой уровень доступа к файлам
- `${AGENT_...}` — ссылки на переменные окружения из Шага 3. Gateway подставит реальные ключи при запуске
- `agentRouting` определяет, как переключаться между агентами в Telegram через команды

**Почему безопасность настроена именно так:**

| Агент | Инструменты | Файлы | Интернет | Обоснование |
|-------|-------------|-------|----------|-------------|
| Ассистент | Нет | Нет | Нет | Минимальная площадь атаки. Если AI будет обманут через текст сообщения — он ничего не сможет сделать, кроме как ответить текстом |
| Кодер | Файлы, команды | Только `/workspaces/coder` | Нет | Может работать с кодом, но только в отведённой папке. Не может выйти в интернет — значит, не может отправить ваши данные наружу |
| Исследователь | Браузер | Нет | Да (через sandbox) | Может искать информацию, но не может сохранять файлы на сервер. Браузер работает внутри Docker-контейнера, изолированно от основной системы |

> **Безопасность читателя:** Агент, работающий с внешним контентом (веб-страницы, документы) — самый уязвимый для атак «косвенной инъекции промптов» (indirect prompt injection — когда вредоносные инструкции спрятаны в содержимом веб-страницы или документа, и AI выполняет их, думая, что это часть задания). Поэтому агенту-исследователю запрещён доступ к файлам памяти и конфигурации — даже если злоумышленник «обманет» его через подготовленную страницу, агент не сможет изменить настройки системы или данные других агентов.

---

**Шаг 5. Защита файлов идентичности агентов**

Файлы `SOUL.md` и `MEMORY.md` определяют «личность» и «память» ваших агентов — инструкции, по которым AI ведёт себя определённым образом. Атака «отравления памяти» (memory poisoning) — это когда вредоносный контент модифицирует эти файлы, создавая постоянный бэкдор (backdoor — скрытый вход), который переживает перезагрузки и работает незаметно.

Защитите эти файлы от изменения:

```bash
# Защищаем файлы идентичности агентов от отравления
sudo chattr +i ~/openclaw/workspace/SOUL.md
sudo chattr +i ~/openclaw/workspace/MEMORY.md
```

Теперь, чтобы обновить файлы идентичности, нужно явно снять защиту (`sudo chattr -i`), внести изменения и поставить защиту обратно — точно так же, как мы делали с конфигурацией SSH в Фазе 2.

---

**Шаг 5.5. Ограничиваем ресурсы контейнеров агентов**

Без ограничений один «зависший» контейнер агента может потребить всю оперативную память сервера и обрушить систему — включая Gateway и остальных агентов. Это как если бы один жилец в многоквартирном доме включил все краны на полную — воды не хватит остальным.

В файле `gateway.yaml` обновите секцию `defaults` в `sandbox`, добавив лимиты ресурсов:

```yaml
sandbox:
  mode: all
  scope: dedicated
  docker: true
  memory: "512m"
  cpus: "0.5"
```

Параметр `memory: "512m"` ограничивает каждый контейнер агента до 512 МБ памяти, а `cpus: "0.5"` — до половины одного ядра процессора.

Расчёт для VPS с 4 ГБ RAM: Gateway (1.5 ГБ) + 3 агента по 512 МБ (1.5 ГБ) + система (0.5 ГБ) = ~3.5 ГБ — укладываемся в лимит с запасом.

### Частые проблемы

| Симптом | Причина | Решение |
|---------|---------|---------|
| Агент не переключается по команде `/coder` или `/research` | Маршрутизация не настроена или Gateway не перечитал конфигурацию | Проверьте секцию `agentRouting` в `gateway.yaml`. Перезапустите: `openclaw gateway restart` |
| Кодер не может записать файлы, ошибка `Permission denied` | Директория `/opt/openclaw/workspaces/coder` принадлежит другому пользователю | Исправьте владельца: `sudo chown openclaw:openclaw /opt/openclaw/workspaces/coder` |
| Исследователь не открывает веб-страницы | Браузер не разрешён в конфигурации sandbox или Docker не может запустить headless-браузер (браузер без графического интерфейса, работающий в фоне) | Проверьте `allow: [browser]` в конфигурации. Пересоздайте sandbox: `openclaw sandbox recreate` |
| На вопрос отвечает не тот агент | Неверно указан `defaultAgent` или маршрутизация путает команды | Проверьте `defaultAgent: assistant` и секцию `agentRouting` в `gateway.yaml` |
| `Error: credential not found` при запуске | Переменные окружения не загружены | Проверьте файл `/etc/openclaw/env` и перезапустите сервис: `openclaw gateway restart` |
| Один из контейнеров не запускается | Docker не хватает ресурсов (памяти или дискового пространства) | Проверьте: `docker ps -a` для статуса контейнеров, `df -h` для свободного места, `free -m` для памяти |
| `security audit` показывает FAIL | Конфигурация содержит небезопасные параметры | Выполните `openclaw security audit --fix` для автоматического исправления безопасных проблем, остальные исправьте вручную по описанию |

## Фаза 6: Обслуживание и мониторинг

### Что мы делаем и зачем

В предыдущих фазах мы установили и настроили OpenClaw на VPS. Но сервер — это не бытовой прибор, который можно «включить и забыть». Без регулярного обслуживания настройки безопасности могут «поплыть» (кто-то случайно изменит конфиг через чат), обновления не установятся, а расходы на API выйдут из-под контроля.

В этой фазе мы настроим простой ритуал обслуживания: 5 минут в неделю и 15 минут в месяц — этого достаточно, чтобы система оставалась защищённой и работала стабильно.

### Схема

```
┌── Еженедельно (5 минут) ────────────────────────────┐
│                                                       │
│  security audit → git diff → cron list                │
│  → проверка перезагрузки → расход токенов             │
│                                                       │
├── Ежемесячно (15 минут) ────────────────────────────┤
│                                                       │
│  openclaw update → apt upgrade                        │
│  → резервная копия → обзор расходов                   │
│                                                       │
├── По необходимости (при сбоях) ─────────────────────┤
│                                                       │
│  logs → doctor → restart → rollback конфига           │
│                                                       │
└───────────────────────────────────────────────────────┘
```

### Шаги

#### 1. Еженедельная проверка (5 минут)

Это ваш «обход» сервера — как обход здания охранником. Пять команд, которые дадут полную картину состояния системы. Выполняйте их каждую неделю в удобный день.

**Шаг 1.1. Аудит безопасности**


Эта команда запускает более 50 автоматических проверок: права доступа к файлам, настройки каналов, изоляцию контейнеров, сетевые подключения. Флаг `--deep` добавляет ещё и живую проверку WebSocket-соединения (протокол для обмена данными в реальном времени между шлюзом и клиентами).

Ожидаемый вывод — все проверки пройдены:

Если какие-то проверки не прошли — команда покажет, что именно не так, и предложит исправление. Можно применить безопасные исправления автоматически:

Эта команда исправит только те проблемы, которые можно безопасно починить автоматически, и выведет список оставшихся.

#### 1.5. Ежедневный автоматический аудит (настраивается один раз)

Вместо того чтобы вручную проверять сервер каждый день, настроим автоматический ночной аудит — скрипт, который проверяет ключевые метрики безопасности и записывает отчёт в лог-файл. Это как видеонаблюдение: запись идёт круглосуточно, а вы просматриваете её только если что-то случилось. Подход основан на рекомендациях SlowMist по аудиту безопасности.

Создайте файл скрипта:

```bash
sudo mkdir -p /opt/security
sudo nano /opt/security/nightly-audit.sh
```

Вставьте содержимое:

```bash
#!/bin/bash
# /opt/security/nightly-audit.sh
# Ночной аудит безопасности

REPORT="/var/log/security-audit/$(date +%Y-%m-%d).log"
mkdir -p /var/log/security-audit

echo "=== Аудит $(date) ===" > "$REPORT"

# 1. Новые открытые порты
ss -tlnp >> "$REPORT"

# 2. Изменения в /etc
cd /opt/security/baselines 2>/dev/null && \
  find /etc -type f -exec sha256sum {} + 2>/dev/null | \
  diff etc_hashes.baseline - >> "$REPORT" 2>/dev/null

# 3. Проверка безопасности OpenClaw
openclaw security audit --deep >> "$REPORT" 2>/dev/null

# 4. Проверка конфигурации
cd ~/.openclaw && git diff >> "$REPORT"

# 5. Использование памяти Gateway
ps -o rss=,vsz=,pid=,comm= -C node >> "$REPORT"

echo "=== Завершено ===" >> "$REPORT"
```

Установите скрипт и добавьте в расписание cron (планировщик задач — программа, которая автоматически запускает команды по расписанию, как будильник для скриптов):

```bash
sudo chmod 700 /opt/security/nightly-audit.sh
echo "0 3 * * * root /opt/security/nightly-audit.sh" | sudo tee /etc/cron.d/security-audit
```

Скрипт будет запускаться каждую ночь в 3:00 и записывать отчёт в `/var/log/security-audit/`. Просматривать отчёты можно командой:

```bash
cat /var/log/security-audit/$(date +%Y-%m-%d).log
```

#### 1.6. Безопасность навыков (Skills)

Навыки (Skills) — это дополнительные возможности, которые можно устанавливать из каталога ClawHub (как приложения из магазина).

В январе-феврале 2026 года произошла массовая атака на цепочку поставок ClawHub, получившая название **ClawHavoc** — более 800 вредоносных навыков (около 20% всего каталога) содержали трояны (программы, маскирующиеся под полезные), кейлоггеры (перехватчики нажатий клавиш) и стиллеры (программы для кражи паролей и токенов). Многие из этих навыков были точными копиями популярных, с минимальными изменениями в названии.

Правила безопасности при работе с навыками:

1. Никогда не устанавливайте навыки без предварительного аудита исходного кода — просмотрите файлы навыка на GitHub перед установкой
2. Фиксируйте версии: `clawhub install <skill> --version X.Y.Z` — это гарантирует, что навык не обновится автоматически на вредоносную версию
3. Не выполняйте команды из раздела "Prerequisites" в SKILL.md — вредоносные навыки часто просят выполнить опасные команды под видом «подготовки к установке»
4. Используйте ClawVet для сканирования навыков перед установкой: `github.com/MohibShaikh/clawvet`

**Автоматическое сканирование навыков**

Ручной аудит полезен, но недостаточен при масштабе кампании ClawHavoc (800+ вредоносных навыков). Используйте автоматические сканеры:

| Инструмент | Что делает | Установка |
|-----------|-----------|-----------|
| Cisco AI Skill Scanner | Статический + семантический анализ + VirusTotal | `pip install cisco-ai-skill-scanner` |
| Aguara | 177 правил обнаружения, не требует API-ключей | github.com/garagon/aguara |

Пример сканирования перед установкой навыка:

```bash
skill-scanner scan /path/to/skill --policy strict
```

Если сканер находит проблемы уровня `high` или `critical` — не устанавливайте навык.

#### 3. Мониторинг расходов

AI-модели тарифицируются по количеству токенов — чем больше текста отправляете и получаете, тем дороже. Без контроля расходы могут неприятно удивить в конце месяца.

**Шаг 3.1. Проверка расходов на стороне провайдера**

Данные `openclaw status` — это оценка. Точные цифры смотрите в личных кабинетах провайдеров:

- **Anthropic (Claude):** https://console.anthropic.com — раздел «Usage»
- **OpenAI (GPT):** https://platform.openai.com/usage — раздел «Usage»

Заходите туда хотя бы раз в месяц.

**Шаг 3.2. Установка лимитов расходов**

Обязательно установите лимиты трат у каждого провайдера — это страховка от неожиданных расходов:

- **Anthropic:** https://console.anthropic.com → Settings → Spending Limits
- **OpenAI:** https://platform.openai.com/settings → Billing → Usage limits

Рекомендуемые начальные лимиты для личного использования: $10-20 в месяц на каждого провайдера. Позже скорректируете по реальному расходу.

**Шаг 3.3. Зачем отдельные API-ключи для каждого агента**

В Фазе 4 мы настраивали отдельный API-ключ для каждого агента. Это не только мера безопасности — это ещё и инструмент контроля расходов. Если у каждого агента свой ключ, вы видите в панели провайдера, какой именно агент сколько тратит. И если один агент выходит из-под контроля, вы можете отключить только его ключ, не затрагивая остальных.

**Шаг 3.4. Примерные ежемесячные расходы**

Ориентировочные расходы при типичном личном использовании (10-30 сообщений в день):

#### 4. Резервное копирование

Если сервер выйдет из строя, вам нужно будет восстановить конфигурацию. Всё важное хранится в одном каталоге: `~/.openclaw/`.

**Шаг 4.1. Что нужно копировать**

Каталог `~/.openclaw/` содержит:
- `openclaw.json` — главный файл конфигурации
- Ключи и токены каналов (Telegram, WhatsApp)
- Настройки агентов
- Историю сессий

Этот каталог уже под контролем версий git (мы настроили это в Фазе 3), поэтому у вас есть история всех изменений на самом сервере. Но если сервер пропадёт — пропадёт и эта история. Поэтому нужна копия на вашем компьютере.

**Шаг 4.2. Копирование на ваш компьютер**

Выполните эту команду **на вашем локальном компьютере** (не на сервере):

```bash
scp -r openclaw@ваш-сервер:~/.openclaw/ ~/openclaw-backup-$(date +%Y-%m-%d)/
```

Команда `scp` (Secure Copy — безопасное копирование) скачивает весь каталог `~/.openclaw/` с сервера в папку на вашем компьютере с датой в имени.

Ожидаемый вывод:

```
openclaw.json                     100%  2.1KB   125.3KB/s   00:00
...
```

**Шаг 4.3. Простой скрипт для регулярных копий**

Чтобы не набирать команду вручную каждый раз, создайте на своём компьютере файл-скрипт (набор команд, который можно запускать одним действием):

```bash
#!/bin/bash
BACKUP_DIR="$HOME/openclaw-backups"
DATE=$(date +%Y-%m-%d)
mkdir -p "$BACKUP_DIR"
scp -r openclaw@ваш-сервер:~/.openclaw/ "$BACKUP_DIR/backup-$DATE/"
echo "Копия сохранена в $BACKUP_DIR/backup-$DATE/"

# Удаляем копии старше 90 дней
find "$BACKUP_DIR" -maxdepth 1 -type d -name "backup-*" -mtime +90 -exec rm -rf {} +
```

Сохраните его как `~/openclaw-backup.sh` и сделайте исполняемым:

```bash
chmod +x ~/openclaw-backup.sh
```

Запускайте раз в месяц:

```bash
~/openclaw-backup.sh
```

Ожидаемый вывод:

```
Копия сохранена в /home/вашеимя/openclaw-backups/backup-2026-03-15/
```

**Шаг 4.4. Автоматическое резервное копирование памяти агентов**

Помимо полной копии конфигурации, нужно отдельно сохранять «мозг» агентов — файлы памяти. Если память агента будет отравлена (через атаку memory poisoning), вы сможете откатить её к чистой версии из резервной копии.

Создайте скрипт:

```bash
sudo nano /opt/security/agent-brain-backup.sh
```

Вставьте содержимое:

```bash
#!/bin/bash
# /opt/security/agent-brain-backup.sh
BACKUP_DIR="/opt/backups/openclaw-brain"
AGENT_DIR="$HOME/.openclaw"
mkdir -p "$BACKUP_DIR"

cd "$BACKUP_DIR"
[ -d .git ] || git init

cp -r "$AGENT_DIR/memory/" "$BACKUP_DIR/memory/" 2>/dev/null
cp "$AGENT_DIR/openclaw.json" "$BACKUP_DIR/" 2>/dev/null

git add -A
git diff --cached --quiet || git commit -m "backup $(date +%Y-%m-%d_%H%M)"
```

Установите скрипт и добавьте в расписание (каждый час):

```bash
sudo chmod 700 /opt/security/agent-brain-backup.sh
echo "0 * * * * openclaw /opt/security/agent-brain-backup.sh" | sudo tee /etc/cron.d/agent-backup
```

Если вам потребуется восстановить память агента (например, после обнаружения подозрительного поведения):

```bash
cd /opt/backups/openclaw-brain
git log --oneline -10
git checkout <commit> -- memory/
cp -r memory/ ~/.openclaw/memory/
openclaw gateway restart
```

Замените `<commit>` на хеш нужного коммита из списка — например, последний «чистый» снимок до того, как агент начал вести себя странно.

#### 5. Обновление сервера

Операционная система сервера тоже нуждается в обновлениях. Автоматические обновления безопасности установлены в Фазе 2 (пакет `unattended-upgrades`), но полное обновление всех пакетов лучше делать вручную раз в месяц.

**Шаг 5.1. Обновите список пакетов и установите обновления**

```bash
sudo apt update && sudo apt upgrade -y
```

Первая часть (`apt update`) скачивает свежий список доступных пакетов. Вторая (`apt upgrade -y`) устанавливает все обновления. Флаг `-y` означает автоматическое согласие на установку.

Ожидаемый вывод:

```
Reading package lists... Done
Building dependency tree... Done
The following packages will be upgraded:
  libssl3 openssl ...
3 upgraded, 0 newly installed, 0 to remove.
...
```

**Шаг 5.2. Проверьте, нужна ли перезагрузка**

```bash
cat /var/run/reboot-required 2>/dev/null || echo "Перезагрузка не нужна"
```

Если перезагрузка нужна (обычно после обновления ядра), выполните следующий шаг.

**Шаг 5.3. Безопасная перезагрузка сервера**

Перед перезагрузкой убедитесь, что OpenClaw установлен как системный сервис (daemon — фоновый процесс, который автоматически запускается при старте системы). Мы сделали это в Фазе 3 командой `openclaw onboard --install-daemon`. В этом случае после перезагрузки OpenClaw запустится сам.

Проверьте, что сервис настроен на автозапуск:

```bash
systemctl is-enabled openclaw
```

Ожидаемый вывод:

```
enabled
```

Если вывод `enabled` — можно безопасно перезагружать:

```bash
sudo reboot
```

Сервер станет недоступен на 1-2 минуты. После перезагрузки подключитесь снова и проверьте:

```bash
ssh openclaw@ваш-сервер
systemctl status openclaw
```

Ожидаемый вывод:

```
● openclaw.service - OpenClaw Gateway
     Loaded: loaded (/etc/systemd/system/openclaw.service; enabled)
     Active: active (running) since ...
```

Статус `active (running)` означает, что шлюз запустился автоматически после перезагрузки. Ваши каналы (Telegram, WhatsApp) переподключатся сами в течение нескольких секунд.

#### 6. Когда что-то пошло не так

Если OpenClaw перестал отвечать на сообщения, работает с ошибками или ведёт себя странно — вот порядок диагностики, от простого к сложному.

**Шаг 6.1. Проверьте статус сервиса**

```bash
systemctl status openclaw
```

Если статус `active (running)` — сервис работает, проблема в чём-то другом. Если `inactive` или `failed` — сервис упал.

**Шаг 6.2. Прочитайте логи**

```bash
openclaw logs
```

Логи (журнал событий — записи всех действий системы) покажут последние события. Ищите строки с `ERROR` или `WARN` — они указывают на проблемы.

Чтобы увидеть только ошибки:

```bash
openclaw logs | grep -i error
```

Типичные ошибки и их значение:

| Ошибка в логах | Что означает | Что делать |
|----------------|-------------|-----------|
| `ANTHROPIC_API_KEY invalid` | Невалидный или отозванный API-ключ | Обновите ключ в конфигурации |
| `rate limit exceeded` | Превышен лимит запросов к провайдеру | Подождите или увеличьте лимит у провайдера |
| `channel disconnected` | Потеряно соединение с Telegram/WhatsApp | Перезапустите шлюз (шаг 6.4) |
| `sandbox: docker not running` | Docker не запущен | `sudo systemctl start docker` |
| `ENOSPC` | Закончилось место на диске | Освободите место: `sudo apt autoremove && sudo docker system prune` |

**Шаг 6.3. Автоматическая диагностика**

```bash
openclaw doctor
```

Команда проверяет все компоненты системы: версию Node.js, Docker, состояние шлюза, подключение каналов, работу песочниц. Если что-то не так — покажет проблему и предложит решение.

**Шаг 6.4. Перезапуск шлюза**

Если проблема не решается — перезапустите шлюз:

```bash
openclaw gateway restart
```

Ожидаемый вывод:

```
Stopping gateway... done
Starting gateway... done
Gateway is running
```

Перезапуск занимает несколько секунд. Каналы переподключатся автоматически.

Если `openclaw gateway restart` не помогает, попробуйте через systemd:

```bash
sudo systemctl restart openclaw
```

**Шаг 6.5. Проверка состояния сервиса**

```bash
systemctl status openclaw
```

Убедитесь, что статус `active (running)` и нет ошибок в последних строках вывода.

**Шаг 6.6. Откат конфигурации (крайняя мера)**

Если проблема появилась после изменения настроек и вы не можете понять, что именно сломалось — верните конфигурацию к последнему рабочему состоянию:

```bash
cd ~/.openclaw && git checkout .
```

Эта команда отменит все изменения конфигурации, сделанные после последнего `git commit`. Затем перезапустите шлюз:

```bash
openclaw gateway restart
```

Если нужно откатиться к конкретному сохранению — посмотрите историю:

```bash
cd ~/.openclaw && git log --oneline
```

Ожидаемый вывод:

```
abc1234 before update 2026-03-15
def5678 before update 2026-02-15
789abcd baseline hardened config
```

Откатитесь к нужной точке:

```bash
cd ~/.openclaw && git checkout <хеш-коммита> -- .
openclaw gateway restart
```

Замените `<хеш-коммита>` на нужный идентификатор из списка (например, `789abcd`).

### Частые проблемы

**Проблема:** `openclaw security audit` показывает предупреждения, которых раньше не было.

Причина: конфигурация изменилась — возможно, после обновления OpenClaw добавились новые проверки, или кто-то изменил настройки.

Решение:
```bash
openclaw security audit --fix
cd ~/.openclaw && git diff
```
Посмотрите, что изменилось, и если изменения корректны — сохраните:
```bash
cd ~/.openclaw && git add . && git commit -m "post-audit fix $(date +%Y-%m-%d)"
```

---

**Проблема:** `git diff` показывает изменения, которые вы не делали.

Причина: агент или обновление изменили конфигурацию.

Решение: внимательно прочитайте, что именно изменилось. Если изменения нежелательные:
```bash
cd ~/.openclaw && git checkout .
openclaw gateway restart
```

---

**Проблема:** после `sudo reboot` сервер не отвечает более 5 минут.

Причина: проблема с загрузкой ОС или сетью.

Решение: зайдите в панель управления вашего VPS-провайдера (Hetzner, DigitalOcean и т.д.) и воспользуйтесь веб-консолью (виртуальный монитор сервера, доступный через браузер). Она покажет, на каком этапе загрузки застрял сервер.

---

**Проблема:** `openclaw update` завершается с ошибкой.

Причина: проблема с сетью или конфликт зависимостей.

Решение:
```bash
openclaw doctor
sudo apt update && sudo apt upgrade -y
openclaw update
```
Если не помогает — проверьте, достаточно ли места на диске:
```bash
df -h /
```
Свободного места должно быть не менее 2 ГБ.

---

**Проблема:** каналы (Telegram/WhatsApp) отключились после перезагрузки.

Причина: обычно каналы переподключаются автоматически за 10-30 секунд. Если прошло больше минуты — сессия могла истечь.

Решение:
```bash
openclaw channels status
```
Если канал показывает статус `disconnected`:
```bash
openclaw channels login <имя-канала>
```
Для WhatsApp потребуется заново отсканировать QR-код.

---

**Проблема:** расход токенов аномально высокий.

Причина: зацикленный агент, спам в канале, или слишком длинный контекст диалогов.

Решение:
```bash
openclaw logs | grep -i "tokens"
openclaw status --all
```
Проверьте, какой агент расходует больше всего. При необходимости временно отключите проблемный канал:
```bash
openclaw channels remove <имя-канала>
```

## Что дальше?

Поздравляем — у вас работает полностью настроенный, защищённый личный AI-ассистент на собственном сервере. Ваши данные под вашим контролем, система защищена пятью уровнями безопасности, а обслуживание занимает 5 минут в неделю.

Вот что можно сделать дальше, когда освоитесь с базовой настройкой:

**Добавить WhatsApp-канал**

```bash
openclaw channels add whatsapp
```

**Настроить автоматические задачи**

Запланируйте регулярные задачи, которые агент будет выполнять сам. Например, ежедневная сводка новостей в 9 утра:

```bash
openclaw cron add --schedule "0 9 * * *" --agent assistant --prompt "Подготовь краткую сводку главных новостей за вчера"
```

Другие идеи: напоминания, отчёты по расходам, мониторинг веб-страниц на изменения.

**Попробовать разные AI-модели**

OpenClaw поддерживает модели разных провайдеров. Попробуйте создать второго агента с другой моделью и сравните качество ответов:

- Claude Sonnet — хороший баланс скорости и качества
- Claude Opus — максимальное качество для сложных задач
- GPT-4o — альтернатива от OpenAI
- Модели с открытым исходным кодом через совместимые API

**Исследовать дополнительные роли агентов**

В Фазе 4 мы настроили базовых агентов. По мере необходимости можно добавить:

- **scheduler** — управление расписанием и напоминаниями
- **notifier** — отправка уведомлений по событиям
- **researcher** — поиск информации в интернете (в изолированном браузере)
- **coder** — помощь с программированием (с доступом к файловой системе в песочнице)

**Рассмотреть альтернативные архитектуры**

Если ваши требования к безопасности вырастут — например, для работы с финансовыми данными или в регулируемых отраслях — рассмотрите альтернативы:

- **DigitalOcean App Platform** — управляемая платформа с контейнерной изоляцией «из коробки», автоматическим масштабированием и предсказуемой стоимостью. Подходит, если вы не хотите заниматься администрированием сервера
